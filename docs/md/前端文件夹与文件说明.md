# 前端文件夹与文件说明

> 说明 `frontend/` 下每个文件夹、每个文件的作用，以及为什么要这样设置。

---

## 一、整体结构一览

```
frontend/
├── index.html              # 浏览器入口 HTML
├── package.json            # 依赖与脚本
├── package-lock.json       # 锁定依赖版本（一般不手改）
├── vite.config.js         # Vite 构建/开发配置
├── eslint.config.js       # 代码检查规则
├── public/                 # 静态资源，原样复制到构建结果
│   ├── vite.svg
│   └── default-avatar.svg
└── src/                    # 源码（主要在这里开发）
    ├── main.jsx            # React 挂载入口
    ├── App.jsx             # 根组件（路由 + 布局）
    ├── App.css              # 根组件样式
    ├── index.css            # 全局样式
    ├── api/                 # 与后端 API 通信的封装
    ├── components/          # 可复用的 UI 组件
    ├── context/             # React 全局状态（如登录态）
    ├── data/                # 前端 mock 数据（开发/占位用）
    └── pages/               # 页面级组件（一个路由一页）
```

---

## 二、根目录下的文件（frontend/）

### 1. `index.html`

**作用**：浏览器真正请求的第一个文件。里面有一个 `<div id="root">`，以及一行 `<script type="module" src="/src/main.jsx">`，用来加载 React 入口。

**为什么要单独放根目录**：Vite 规定入口 HTML 必须在项目根目录，这样 `npm run dev` 时访问的 `/` 会返回这个 HTML，再由它去加载 `main.jsx`，进而挂载整个 React 应用。

---

### 2. `package.json`

**作用**：定义项目名称、依赖（react、react-dom、react-router-dom）和脚本（`dev`、`build`、`lint`、`preview`）。`npm install` 根据它装包，`npm run dev` 执行里面配置的 `vite`。

**为什么要这样设置**：用 npm 统一管理依赖和命令，团队和部署环境都用同一套脚本和版本。

---

### 3. `vite.config.js`

**作用**：Vite 的配置文件。当前做了两件事：启用 React 插件（解析 JSX）、把开发服务器设成 `host: true`（方便用电脑 IP 在手机访问）。

**为什么要单独一个配置文件**：开发/构建行为（端口、代理、环境变量、是否开 sourcemap 等）都集中在这里，改配置不用动业务代码。

---

### 4. `eslint.config.js`

**作用**：ESLint 的规则配置，用来做代码检查（未用变量、格式等）。`npm run lint` 会跑它。

**为什么要这样设置**：统一代码风格、提前发现明显错误，和编辑器配合使用。

---

## 三、`public/` 文件夹

**文件夹作用**：放「不经过打包、原样复制到构建结果」的静态文件。部署后访问 `/vite.svg`、`/default-avatar.svg` 就是这里的文件。

| 文件 | 作用 |
|------|------|
| `vite.svg` | Vite 默认的 favicon/图标，可后续换成自己项目的 logo。 |
| `default-avatar.svg` | 用户未设置头像时显示的默认头像，在 MyZone、PostCard、ProfileEdit 等地方用 `<img src="/default-avatar.svg" />` 引用。 |

**为什么要用 public**：这类文件路径固定、不需要打包或改名字，放 public 后直接用根路径引用即可，和 `src` 里需要 import 的素材区分开。=========??????????===========

## 四、`src/` 下的入口与全局文件

### 1. `main.jsx`

**作用**：React 的真正入口。用 `createRoot(document.getElementById('root')).render(<App />)` 把根组件挂到 HTML 里的 `#root` 上，并包一层 `StrictMode` //?????方便开发时发现问题。同时引入 `index.css` 作为全局样式。

**为什么入口叫 main**：约定俗成，Vite/React 模板通常用 `main.js` 或 `main.jsx` 作为单页应用的 JS 入口，在 `index.html` 里只引用这一个脚本即可。

---

### 2. `App.jsx`

**作用**：根组件。负责声明整站路由（BrowserRouter、Routes、Route）、用 AuthProvider 包住全局、用 AuthGuard 保护需要登录的区域、用 Layout 包住子路由，并定义每个 path 对应哪个页面组件。详见《逐行解析 App.jsx》。

**为什么路由都写在这里**：路由是「整站级」的配置，放在根组件里集中管理，一目了然；子页面只关心自己的内容，不关心 URL 怎么配。

---

### 3. `App.css`

**作用**：根组件级别的样式。可以放全站共用的基础样式（如重置、字体、主题变量等），目前和 `index.css` 可能有重叠，实际项目里可以二选一或分工（例如 index 只做 reset，App 做布局相关）。

**为什么和 App 同名的 css**：和 `App.jsx` 配套，便于「改 App 时顺带改样式」；用 Vite 时在 `App.jsx` 里 `import './App.css'` 即可。

---

### 4. `index.css`

**作用**：全局样式。在 `main.jsx` 里被引入，所以会作用到整个应用。常用来写 CSS 变量、body 背景、默认字体等。

**为什么单独一个 index.css**：和「入口」对应，表示「最先加载的全局样式」，和各个页面/组件的局部样式分开，避免把全站样式塞进某个组件里。

---

## 五、`src/api/` —— 与后端通信

**文件夹作用**：集中放「调后端接口」的代码。页面里不直接写 `fetch(某 URL)`，而是调用这里封装好的函数（如 `getPostList()`、`createPost()`），这样接口地址、错误处理、token 携带都可以在一处维护。

| 文件 | 作用 |
|------|------|
| `config.js` | 定义 `API_BASE_URL`（如本地 `http://127.0.0.1:4040`），供下面所有 api 文件用。根据是否本地环境或环境变量决定用哪个地址。 |
| `posts.js` | 帖子相关接口：getPostList、getPostDetail、createPost、deletePost、toggleLike、getPostComments、createComment、deleteComment。每个函数内部用 `fetch(API_BASE_URL + '/api/posts/...')` 并处理 `status`、token。 |
| `canteen.js` | 食堂相关接口：getProductComments（商品评论列表）、postProductComment（发表商品评论）。和 `posts.js` 一样用 config 的 base URL。 |

**为什么要单独一个 api 文件夹**：  
- **接口集中**：改后端地址或返回格式时，只改 api 层，不用去每个页面找 fetch。  
- **复用**：多个页面可能调同一个接口（如帖子列表），写一次即可。  
- **职责清晰**：页面只负责「调哪个函数、展示什么」，不负责「请求怎么写、错误怎么转」。

---

## 六、`src/components/` —— 可复用 UI 组件

**文件夹作用**：放被多个页面共用、或结构上可拆出来的 UI 块。它们通常不直接对应一个路由，而是被页面「引用」。

| 文件 | 作用 |
|------|------|
| `AuthGuard.jsx` | 路由守卫组件：若未登录且未「暂不登录」，渲染 `<Navigate to="/login" />`，否则渲染 `children`。被 App.jsx 用来包住 Layout。 |
| `Layout.jsx` | 整体布局：顶栏（TopBar）+ 主体（Outlet）+ 底栏（TabBar）。根据当前 pathname 算标题、是否显示返回。所有「在 Layout 下的」页面都共享这一套壳。 |
| `Layout.css` | Layout 的样式。 |
| `TopBar.jsx` | 顶栏：标题、返回按钮、信箱图标等。接收 props 如 title、showBack。 |
| `TopBar.css` | 顶栏样式。 |
| `TabBar.jsx` | 底部 Tab：首页 / 食堂 / 关于 / 我的 / 信箱，用 Link 连到对应路由，并根据当前路径高亮。 |
| `TabBar.css` | 底栏样式。 |
| `PostCard.jsx` | 帖子卡片：头像、用户名、内容摘要、点赞/评论数，点击跳到帖子详情。被 TreeHole、MyPosts 等列表页复用。 |
| `PostCard.css` | 卡片样式。 |

**为什么要单独 components**：  
- **复用**：PostCard 在树洞和「我的帖子」都能用；Layout/TopBar/TabBar 被所有子页面共用。  
- **可读性**：页面文件只写「这一页的逻辑和结构」，把通用块拆到 components，页面更短、更易读。  
- **维护**：改顶栏/底栏/卡片样式时，只改对应组件，不用在多个页面重复改。

---

## 七、`src/context/` —— 全局状态

**文件夹作用**：放用 React Context 提供的「跨很多组件都要用」的状态，避免一层层 props 传递。

| 文件 | 作用 |
|------|------|
| `AuthContext.jsx` | 定义 AuthProvider 和 useAuth()。提供：user、token、isLoggedIn、login、logout、skipLogin、hasSkippedLogin、displayName、displayAvatar、updateProfile、clearProfile。登录态和 token 会存 localStorage，刷新不丢。App 用 `<AuthProvider>` 包住整棵树，所以任意子组件都能 `useAuth()` 取登录态或调 login/logout。 |

**为什么要用 context 单独放一个文件夹**：  
- **跨页面**：登录态在 Login、Layout、PostNew、MyZone、api 请求里都要用，不适合用 props 一层层传。  
- **集中**：所有「全局状态」类的东西（以后若有主题、语言等）也可以放 context，和「只在本页用」的 useState 区分开。

---

## 八、`src/data/` —— Mock 数据

**文件夹作用**：放前端写死的假数据，用于「后端接口还没接好」或「先做静态演示」时，让页面能先跑起来、先调样式和交互。

| 文件 | 作用 |
|------|------|
| `mockPosts.js` | 导出 MOCK_POSTS（假帖子列表）、getPostsByAuthorId(authorId)、getAuthor(authorId)。TreeHole 用 MOCK_POSTS 渲染列表，PostDetail 用 getAuthor，MyPosts 用 getPostsByAuthorId。 |
| `mockComments.js` | 导出 getCommentsForPost(postId)，返回某帖子的假评论。PostDetail 用来看评论区和回复样式。 |
| `mockNotifications.js` | 导出 MOCK_NOTIFICATIONS、getPostPreview(postId)。Mailbox 用它们展示假的通知列表和帖子摘要。 |

**为什么要单独 data 且叫 mock**：  
- **和真实接口区分**：接真实 API 时，把页面里 `import ... from '../data/mockPosts'` 改成 `import { getPostList } from '../api/posts'` 即可，mock 文件可以保留或删掉。  
- **集中**：假数据集中在一个目录，方便以后删或改成从本地 JSON 读，不会散落在各个页面里。

---

## 九、`src/pages/` —— 页面（一个路由一页）

**文件夹作用**：每个文件对应「一个路由」要渲染的整页内容。和 components 的区别是：pages 是「整页」，components 是「页里的某一块」。

| 文件 | 对应路由 | 作用 |
|------|----------|------|
| `Login.jsx` + `Login.css` | `/login` | 登录页：学号/邮箱 + 密码、登录、暂不登录、跳转注册。 |
| `Register.jsx` + `Register.css` | `/register` | 注册页：邮箱/用户名/密码、提交后调 /api/auth/register。 |
| `TreeHole.jsx` + `TreeHole.css` | `/` | 树洞首页：两列帖子列表（当前 mock）、右下角 FAB 跳发帖。 |
| `PostNew.jsx` + `PostNew.css` | `/post/new` | 发帖页：内容 + 最多 3 张图，提交（当前未接 API）。 |
| `PostDetail.jsx` + `PostDetail.css` | `/post/:id` | 帖子详情：正文、作者、点赞、评论列表与发表（当前 mock）。 |
| `Eat.jsx` | `/eat` | 食堂页：占位「内容开发中」。 |
| `AboutUs.jsx` | `/about` | 关于我们：占位。 |
| `MyZone.jsx` + `MyZone.css` | `/myzone` | 我的：头像、用户名、邮箱、入口（修改资料、我的帖子、我的点评、本周点评数）。 |
| `MyPosts.jsx` + `MyPosts.css` | `/myzone/posts` | 我的帖子：当前用 mock 按 user.id 过滤的列表。 |
| `MyReviews.jsx` + `MyReviews.css` | `/myzone/reviews` | 我的点评：占位「功能开发中」。 |
| `ProfileEdit.jsx` + `ProfileEdit.css` | `/myzone/profile` | 修改资料：头像、用户名，当前只写本地 profile，未调后端。 |
| `Mailbox.jsx` + `Mailbox.css` | `/mailbox` | 信箱：当前用 mock 通知列表，点进帖子。 |

**为什么页面都放 pages、且很多带同名 css**：  
- **按路由分文件**：一个路由一个文件，找「登录页逻辑」直接打开 Login.jsx，不会和别的页混在一起。  
- **同名 css**：和「一个页面一个样式文件」对应，样式和页面放在一起，便于维护；Vite 在对应 jsx 里 `import './Login.css'` 即可。

## 十一、为什么要这样设置（总结）

| 设计 | 原因 |
|------|------|
| **api 单独一层** | 接口集中、易改 baseURL 和错误处理、页面只调函数不写 fetch。 |
| **components 和 pages 分开** | 页面 = 一个路由一整页；组件 = 可复用的一块 UI，避免页面文件过大、逻辑混杂。 |
| **context 单独目录** | 全局状态（登录态）集中提供，避免 props 层层传递。 |
| **data 放 mock** | 先跑通 UI 和交互，接真实 API 时只改调用处，不动页面结构。 |
| **每个页面自带同名 css** | 样式和页面一一对应，改某页样式时不会误改别的页。 |
| **public 放静态资源** | 不经过打包、用根路径引用，适合 favicon、默认头像等。 |
| **根目录放配置** | index.html、package.json、vite.config.js、eslint 等和「构建/运行」相关，和业务代码（src）分开，结构清晰。 |

整体上，这是常见的 **按功能/职责分目录** 的 React 前端结构：入口与配置在根目录，业务在 `src`，再按「接口 / 组件 / 全局状态 / 假数据 / 页面」分文件夹，方便协作和后续扩展（例如加 `hooks/`、`utils/`、`styles/` 等）。
