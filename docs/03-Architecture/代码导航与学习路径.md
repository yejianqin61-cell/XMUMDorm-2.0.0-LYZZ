# Jack 项目代码学习顺序

> 目标：理解整个程序的前后端架构与细节，便于自己排查问题、判断 AI 生成代码是否合理，而不是全盘接受。

建议按下面顺序阅读和动手，每步都「先看再想」：这段代码在干什么、和谁配合、出错时可能在哪一环。

---

## 一、先建立「整体地图」

### 1.1 项目是什么、怎么跑起来

- **后端入口**：`server.js`  
  - 用 Express 起一个 HTTP 服务（默认端口 4040）。  
  - 挂载了 CORS、`express.json()`、静态资源 `/uploads`，以及**错误处理中间件**。  
  - 把所有 API 按模块挂到 `/api/xxx`：`/api/auth`、`/api/posts`、`/api/canteen`、`/api/notifications`、`/api/users`。  
- **前端入口**：`frontend/src/main.jsx` → 挂载 `App.jsx`。  
- **前端路由与布局**：`frontend/src/App.jsx`  
  - 用 `BrowserRouter` + `Routes`。  
  - 未登录保护：除 `/login`、`/register` 外，其余页面包在 `AuthGuard` 里；需要登录（或「暂不登录」）才能看到 `Layout`。  
  - `Layout` 里用 `Outlet` 渲染子路由（树洞、食堂、我的、帖子详情等）。

**你要搞清**：  
- 浏览器访问的「前端」和「后端」分别是哪两个进程、端口多少。  
- 前端如何知道后端地址：`frontend/src/api/config.js` 里的 `API_BASE_URL`（本地一般是 `http://127.0.0.1:4040`）。

---

### 1.2 数据流：从前端到数据库

记住这条链子，后面所有功能都是在这条链上某一段：

```
用户操作 → 前端页面/组件 → 前端 API 封装（api/*.js）→ HTTP 请求 
  → 后端路由（routes/*.js）→ 中间件（如 auth、upload）→ 数据库（database.js + MySQL）
  → 返回 JSON → 前端解析 → 更新界面
```

- **后端如何访问数据库**：`database.js` 用 `mysql2` 建连接池，对外提供 `query(sql, params)`。所有路由里查表、改表都是通过这个 `query`。  
- **前端如何发请求**：不直接写 `fetch` 散落在页面里，而是封装在 `frontend/src/api/` 下（如 `posts.js`、`canteen.js`），统一用 `API_BASE_URL` 和可选的 `Authorization: Bearer <token>`。

学习时每看到一个接口，都自己在心里走一遍这条链：从按钮点击到哪段 API、对应后端哪条路由、最后打到哪张表。

## 二、认证与权限（搞懂再去看业务）

### 2.1 后端：登录与 JWT

- **路由**：`routes/auth.js`  
  - 注册、登录、验证码（预留）等。  
  - 登录成功会生成 JWT，把 `userId`、`role` 等塞进 payload，用 `JWT_SECRET` 签名，返回给前端。  
- **中间件**：`middleware/auth.js`  
  - `authenticateToken`：从请求头 `Authorization: Bearer <token>` 里取出 token，用 `jwt.verify` 校验；成功则把解码后的用户写到 `req.user`，后续路由就能用 `req.user.id`、`req.user.role`。  
  - 未带 token 或 token 无效会直接 401/403 返回，不会进入业务逻辑。

你要能回答：**为什么发帖、删帖、点赞等接口需要登录？** 因为对应路由里挂了 `authenticateToken`，只有校验通过才有 `req.user`。

### 2.2 前端：登录状态与请求头

- **状态从哪来**：`frontend/src/context/AuthContext.jsx`  
  - 用 React Context 存 `user`、`token`、`isLoggedIn`，以及 `login`、`logout`。  
  - token 和 user 会持久化到 `localStorage`，刷新页面后仍可恢复登录态。  
- **谁在用**：  
  - `AuthGuard`：未登录且未「暂不登录」就重定向到 `/login`。  
  - 所有需要带 token 的请求：在 `api/posts.js` 等里用 `getAuthHeaders(token)` 把 `Authorization: Bearer <token>` 带上。

你要能回答：**前端发「我的帖子」或「发布帖子」时，token 从哪来？** 从 `useAuth()` 取到的 `token`，在调用 `getPostList({ token })` 或 `createPost(token, body)` 时传进去。

## 三、按「功能模块」顺藤摸瓜

建议一次只盯一个业务模块，把「前端页面 → 前端 API → 后端路由 → 数据库」走通。

### 3.1 帖子系统（树洞）

- **前端**  
  - 列表页：`frontend/src/pages/TreeHole.jsx`（当前用 mock 数据 `MOCK_POSTS`，真实版会调 `getPostList`）。  
  - 单卡：`frontend/src/components/PostCard.jsx`。  
  - 发帖页：`frontend/src/pages/PostNew.jsx`。  
  - 详情页：`frontend/src/pages/PostDetail.jsx`。  
- **前端 API**：`frontend/src/api/posts.js`  
  - `getPostList`、`getPostDetail`、`createPost`、`deletePost`、`toggleLike`、`getPostComments`、`createComment`、`deleteComment`。  
- **后端**：`routes/posts.js`  
  - GET/POST/DELETE/PATCH 与 posts、comments、likes 的对应关系（列表、详情、发布、删除、隐藏、点赞、评论的增删）。  
  - 哪些接口用了 `authenticateToken`，哪些没带（如列表、详情可匿名看）。  
- **数据库**：在 `routes/posts.js` 里能看到用到的表：`posts`、`post_images`、`post_likes`、`comments`、`users` 等。若有建表脚本或文档，可对照看字段含义。

学习时建议：  
1. 先看 `posts.js`（后端）里每个路由的 URL、方法、参数和返回的 `data` 结构。  
2. 再看 `api/posts.js`（前端）里每个函数如何调这些 URL、如何传 token、如何把 `data.status !== 0` 转成抛错。  
3. 最后看 TreeHole / PostCard / PostNew / PostDetail 如何调用这些 API、如何展示和报错。

---

### 3.2 食堂 / 点评

- **前端**：`frontend/src/pages/Eat.jsx`，以及可能用到的 `frontend/src/api/canteen.js`。  
- **后端**：`routes/canteen.js`。  
- 同样用「请求链」思路：页面点哪里 → 调哪个 API → 对应后端哪条路由 → 查/改哪张表。

---

### 3.3 用户与个人资料

- **后端**：`routes/users.js`（个人资料、头像等）。  
- **前端**：`MyZone`、`ProfileEdit` 等页面，以及可能存在的 `api/users.js` 或类似封装。  
- 关注：上传头像等会用到 `middleware/upload.js` 和静态资源路径 `/uploads`。

---

### 3.4 通知 / 信箱

- **后端**：`routes/notifications.js`。  
- **前端**：`frontend/src/pages/Mailbox.jsx` 及相关 API。  
- 了解「谁在什么时候往 `notifications` 表插数据」（例如发帖里的公告、点赞、评论通知）。

## 四、公共与边界

### 4.1 布局与导航

- `Layout.jsx`：顶栏（TopBar）、主体（Outlet）、底栏（TabBar）。  
- `TopBar.jsx` / `TabBar.jsx`：标题、返回、Tab 高亮（由路由 path 决定）。  
- 这样你就知道「新加一个页面」要动哪些地方：App 里加 Route、必要时在 Layout 的 TITLE_BY_PATH 或 Tab 配置里加一项。

### 4.2 错误与提示

- 后端：所有接口统一用 `status: 0` 表示成功，`status: -1` 表示业务/服务器错误，并在 `message` 里给原因。  
- 前端：`api/posts.js` 等里会把 `data.status !== 0` 转成 `throw new Error(data.message)`，页面用 try/catch 或 .catch() 捕获后展示「服务器繁忙，请稍后重试」或具体 `err.message`。  
- 建议：在浏览器 Network 里点开一个失败请求，看 Response 的 `status` 和 `message`，再在前端搜这个接口的调用处，看错误是怎么被捕获和展示的。

### 4.3 环境与配置

- 后端：`dotenv` 从 `.env` 读 `PORT`、`JWT_SECRET`、数据库连接等。  
- 前端：`api/config.js` 的 `API_BASE_URL`，以及 Vite 的 `import.meta.env.VITE_API_BASE_URL`（若存在）。  
- 这样部署或换环境时你知道改哪里、前后端如何对齐。

## 五、推荐学习顺序小结（可打勾）

| 顺序 | 内容 | 重点 |
|------|------|------|
| 1 | `server.js` + `App.jsx` + `api/config.js` | 前后端入口、端口、API 根地址 |
| 2 | `database.js` | 后端如何执行 SQL |
| 3 | `routes/auth.js` + `middleware/auth.js` | 登录、JWT、哪些接口要登录 |
| 4 | `context/AuthContext.jsx` + `AuthGuard.jsx` | 前端登录态、路由保护 |
| 5 | `routes/posts.js` 全文 | 帖子相关接口的 URL、参数、返回结构 |
| 6 | `api/posts.js` 全文 | 前端如何调帖子接口、如何带 token、如何抛错 |
| 7 | `TreeHole.jsx` → `PostCard.jsx`，再 `PostNew.jsx`、`PostDetail.jsx` | 页面如何用 API、如何展示和报错 |
| 8 | `Layout.jsx`、`TopBar`、`TabBar` | 布局与导航 |
| 9 | `routes/canteen.js` + 前端 Eat / canteen API | 食堂模块同一条链 |
| 10 | `routes/users.js`、`routes/notifications.js` + 对应前端 | 用户与通知 |

## 六、如何用这套顺序排查问题

- **「帖子列表/详情显示服务器繁忙」**  
  - 先看 Network：是 4xx/5xx 还是 200 但 `status: -1`？  
  - 若是 401/403：多半是 token 没带或过期，看前端是否在调接口时传了 `token`、AuthContext 里是否还有 token。  
  - 若是 500：看后端控制台报错栈，对应到 `routes/posts.js` 里哪一段（数据库错、未捕获异常等）。  
  - 若是 200 且 `status: -1`：看 `message`，再到后端该接口里搜这个文案，看是哪个校验或分支返回的。

- **「AI 改完代码不知道对不对」**  
  - 先确认它动的是「链子」的哪一段（前端页面 / 前端 API / 后端路由 / 中间件）。  
  - 对照上面学习顺序里对应文件：请求方法、URL、参数、返回结构是否和现有约定一致（例如是否带 token、是否用 FormData 传图等）。  
  - 再在本地跑一遍：列表、发帖、详情、点赞、评论，看是否有报错或界面异常。

按这个顺序把「地图」和「帖子一条链」吃透，再扩展到食堂、用户、通知，你就能逐步独立排查和判断改动是否合理了。
