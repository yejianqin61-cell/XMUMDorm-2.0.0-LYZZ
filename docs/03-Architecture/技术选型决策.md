# 技术选型决策

## 各层技术选型

| 层面 | 选型 | 理由 |
|------|------|------|
| 后端框架 | **Express.js** | 项目已用，团队熟悉，轻量灵活 |
| 数据库 | **MySQL** | 项目已用，关系型数据适合社交平台 |
| 鉴权 | **JWT** | 无状态，支持 Web + App 双端 |
| Web 前端框架 | **React 18** | 已用，组件化生态成熟 |
| Web 路由 | **React Router v6** | SPA 路由，支持守卫 |
| 状态管理 | **TanStack Query** | 服务端状态缓存，Web + RN 统一 |
| HTTP 请求 | **fetch**（Web）/ **fetch**（RN） | API 层代码直接复用 |
| RN 框架 | **Expo SDK 52+** | OTA 热更新、内置推送/相机、跨平台 |
| RN 路由 | **Expo Router**（文件系统路由） | 与 Web React Router 结构相似 |
| 样式方案 | **CSS Modules**（Web）/ **StyleSheet**（RN） | 组件级隔离 |
| 液态玻璃效果 | **CSS backdrop-filter**（Web）/ **expo-blur**（RN） | 半透明降级兜底 |
| 图片上传 | **Multer**（后端本地存储） | 非云存储，服务器本地 |
| 推送 | **Web Push API** + **Expo Notifications** | 双通道推送 |
| 测试 | **Jest 30 + Supertest** | 后端集成测试 |

## 不选的方案

| 不选 | 理由 |
|------|------|
| Next.js / Remix | 项目已有独立 SSR 方案，引入成本高 |
| MongoDB / Prisma | 项目已深度绑定 MySQL + 原生 query |
| Redux / Zustand | TanStack Query 已覆盖服务端状态，Context 覆盖 UI 状态 |
| Firebase / Supabase | 已有完整后端，无需 BaaS |
| Fastify / Koa | 迁移成本高，Express 生态成熟 |
