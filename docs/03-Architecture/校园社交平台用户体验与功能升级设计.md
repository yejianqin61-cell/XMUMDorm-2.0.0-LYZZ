## 1. 文档目标

本设计文档用于回答一个核心问题：

> 在当前功能已经较完整的前提下，XMUMDorm 如何进一步提升“校园感、参与感、留存感、完成度”。

本文不是单一页面的 UI 稿说明，而是一个跨产品、交互、视觉、信息架构、后端能力的升级设计。

目标包括：

1. 统一产品体验方向，避免后续继续“功能越来越多，但整体越来越散”。
2. 给出前端 UI、动效、布局的优化原则与改造方向。
3. 给出更符合校园社交平台定位的功能扩展方案。
4. 给出 P0 / P1 / P2 优先级，便于后续拆任务进入开发。

## 2. 当前产品现状判断

结合现有代码、模块结构和测试情况，项目已经具备较强的“功能完整度”：

- 有树洞、广场、社团、食堂、二手、跑腿、课程评价、通知、个人中心等多模块。
- Web 端与 Capacitor 双形态可共用一套前端主工程。
- 后端 Express + MySQL 已支撑主要互动闭环。

但从用户体验角度，当前仍有以下典型问题：

### 2.1 首页更像“模块集合”，不够像“校园入口”

- 用户进入后能看到很多功能入口，但不一定能立刻理解“现在校园里有什么值得看”。
- 信息组织偏模块导向，而不是用户当下需求导向。

### 2.2 视觉语言已初步形成，但全站不够统一

- 一些页面已经很有风格，尤其玻璃态、卡片、树洞、排行榜等。
- 但不同模块之间卡片密度、留白、标题层级、按钮语义、状态区位置还不完全统一。

### 2.3 互动反馈存在，但产品节奏感还不够强

- 点赞、加载、跳转、发表成功等操作有基本反馈。
- 但还缺少更稳定、更细腻的“用户知道自己刚刚完成了什么”的交互反馈。

### 2.4 功能很多，但“校园关系链”还不够突出

- 目前模块丰富，但“同学院、同课程、同社团、同宿舍楼”的弱关系推荐感还不强。
- 与普通兴趣社区相比，校园平台应该更强调“跟我有关的人和事”。

### 2.5 通知和活动能力具备基础，但尚未成为“校园事务中枢”

- 通知大多还是事件触发型。
- 还没有真正形成“提醒、报名、截止、回复、待办、活动”统一的校园消息中心。

## 3. 体验升级的总原则

后续所有优化建议都遵循下面五条原则。

### 3.1 校园感优先于通用社区感

XMUMDorm 不应只是“一个功能很多的社区”，而应让用户感受到：

- 这里是和我同一个校园的人
- 这里发生的是和我现实生活有关的事情
- 我打开它，是为了更方便地参与校园生活

### 3.2 信息优先级优先于功能堆叠

不要继续单纯增加入口，而应先优化：

- 什么信息先出现
- 什么内容应该被折叠
- 什么操作应该在当前上下文中最容易触发

### 3.3 反馈明确，但不过度炫技

动效不是为了“花”，而是为了：

- 帮助用户理解状态变化
- 降低等待焦虑
- 让重要操作更有完成感

### 3.4 一致性优先于局部漂亮

单个页面再好看，如果和其他模块不协调，最终会拉低整体完成度。

要优先统一：

- 卡片系统
- 间距系统
- 标题层级
- 状态区写法
- 按钮语义
- 列表节奏

### 3.5 先做高频主路径，再做锦上添花

优先覆盖以下高频路径：

1. 打开 App / Web
2. 浏览首页内容
3. 进入树洞 / 广场 / 食堂 / 社团
4. 点赞、评论、收藏、报名、发布
5. 接收通知并回到对应页面

## 4. 核心体验升级方向

### 4.1 首页重构为“今日校园”

#### 当前问题

首页目前更接近“模块入口页”。

#### 目标状态

首页应成为用户每天打开后第一眼看到的“校园摘要面板”，优先展示：

- 今天校园里正在发生什么
- 哪些内容值得我立刻点进去
- 哪些事情与我直接相关

#### 建议结构

从上到下建议调整为：

1. 今日校园头部
   - 问候语
   - 当前日期 / 周次 / 天气入口（可选）
   - 快速提醒数量

2. 今日摘要区
   - 热门树洞
   - 今日活动
   - 食堂热榜
   - 未读通知 / 待办提醒

3. 个性化推荐区
   - 我关注的社团新动态
   - 同学院热门内容
   - 我最近看过的内容

4. 模块入口区
   - 树洞
   - 广场
   - 食堂
   - 二手
   - 跑腿
   - 一站式

#### 价值

- 用户更容易理解产品价值。
- 首屏不再是“模块堆叠”，而是“有信息密度的校园面板”。
- 有助于提升日活和回访率。

### 4.2 建立统一卡片系统

#### 当前问题

不同模块卡片已经都有各自风格，但系统性还不够强。

#### 目标状态

全站卡片统一抽象为 4 类：

1. 信息卡
   - 适合帖子、活动、通知摘要
2. 操作卡
   - 适合快捷入口、发布中心、设置项
3. 数据卡
   - 适合排行榜、统计、个人成就
4. 媒体卡
   - 适合树洞图片贴、二手商品、食堂推荐

#### 统一规范建议

- 圆角：统一主卡片圆角梯度，例如 `16 / 20 / 24`
- 间距：统一卡片内边距和卡片间距
- 标题层级：统一 `页面标题 / 模块标题 / 卡片标题 / 辅助文案`
- 阴影：统一轻、中、重三档
- 状态区：统一空态、错误态、加载态摆放方式
- 操作区：点赞、评论、收藏、更多按钮位置尽量固定

#### 设计补充

现有“液态玻璃”风格可以保留，但要控制使用密度：

- 首页关键入口、头部摘要、重点榜单可以用更强玻璃感
- 普通内容列表不要所有卡片都做重玻璃
- 要避免整页都处于高模糊高反光状态，影响阅读效率

### 4.3 建立轻量、明确的动效体系

#### 目标

让用户更明确感知状态变化，而不是简单“点了没反应”。

#### 建议动效分层

##### A. 页面级动效

- 一级模块切换：轻微淡入或滑入
- 二级详情页：从底部或右侧进入
- 返回时保持方向一致

##### B. 列表级动效

- 卡片首屏分批出现
- 新内容插入列表时有轻量过渡
- 下拉刷新结束时有明确完成反馈

##### C. 操作级动效

- 点赞：轻微弹性反馈
- 评论发送成功：输入区域轻量收起 / 清空并提示
- 收藏 / 报名成功：图标状态切换 + 文案反馈
- 发布成功：返回列表时新卡片高亮短暂出现

##### D. 过渡态优化

- Skeleton 尽量拟合真实内容结构
- 异步切换不应白屏闪烁
- 错误反馈应带重试入口

#### 原则

- 动效时长宜短，优先 `120ms ~ 260ms`
- 高频操作不做重动画
- 移动端优先保障稳定与帧率

### 4.4 重新组织内容布局，降低“长页面疲劳”

#### 当前问题

#### 建议

在长列表中增加“节奏变化点”：

- 热门标签
- 今日讨论焦点
- 你可能感兴趣
- 同学院正在看
- 社团推荐
- 二手精选

#### 示例

1. 顶部推荐话题
2. 主内容流
3. 中插“你可能也想看”
4. 内容流继续

这样列表阅读更有呼吸感。

## 8. Web + Capacitor 适配原则

本项目明确以前端 `web + capacitor` 双形态为主，不再以 RN 原生为当前目标。

因此设计时应遵循：

### 8.1 相同信息架构，不同交互细节

- Web 更强调信息密度
- Capacitor 更强调触控舒适度和过渡自然度

### 8.2 同一组件系统，按设备能力降级

- 动效、模糊、阴影要可降级
- 列表滚动、骨架、底部固定条需兼容移动端安全区
- yjq：尤其要注意顶部safearea，不能和状态栏的字粘起来

### 8.3 页面优先级一致

不做“两套产品逻辑”，只做“一套信息架构 + 两端细节适配”。

### 9.4 关系链

## 14. 结论

XMUMDorm 当前最大的机会，不是简单地继续增加模块，而是把它从“功能丰富的校园项目”升级成“真正有校园生活氛围的产品”。

最关键的不是某一个按钮或动画，而是三件事：

1. 首页要从模块拼盘升级为“今日校园”
2. 体验要从局部好看升级为全站统一
3. 功能要从普通社区能力升级为“校园关系链 + 活动闭环 + 事务通知”

如果后续按本文推进，XMUMDorm 会更像一个真正能被学生每天打开、持续使用、并且愿意推荐给同学的校园平台。
