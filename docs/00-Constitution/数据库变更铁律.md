# 数据库变更铁律

## 核心原则

> **任何 Schema 变更必须走 Migration，禁止手动 ALTER TABLE。**

---

## 迁移规则

### 1. 新增迁移

```
migrations/
├── 001_xxx.sql    ← 旧编号体系（前缀 0 对齐）
├── ...
├── 056_admin.sql  ← 当前最新
└── 057_next.sql   ← 新迁移按编号递增
```

| 规则 | 说明 |
|------|------|
| 文件名 | `NNN_<动作>_<对象>.sql`，如 `047_treehole_tag_visibility.sql` |
| 编号 | 三位数字，严格递增，不可重复使用已有编号 |
| 精度 | 一个迁移只做一件事（一张表 / 一组紧密关联的变更） |
| 可重复执行 | 使用 `IF NOT EXISTS` / `IF EXISTS` 保护 |
| 回退脚本 | 在注释中说明回退方式（或单独写 `.down.sql`） |

### 2. 迁移内容规范

```sql
-- migration: 058_add_user_preferences.sql
-- description: 用户偏好设置表
-- depends: 001 (users 表必须存在)
-- reversible: DROP TABLE IF EXISTS user_preferences;

CREATE TABLE IF NOT EXISTS user_preferences (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    preference_key VARCHAR(64) NOT NULL,
    preference_value TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id),
    UNIQUE KEY uk_user_pref (user_id, preference_key)
);
```

### 3. 执行迁移

```bash
# 一键全部迁移（推荐）
npm run migrate:all

# 单条迁移
node scripts/run-migration-058-add-user-preferences.js

# 增量迁移（仅执行未跑过的）
node scripts/run-incremental-migrations.js
```

### 4. init-db.sql

`init-db.sql` 是全新数据库的初始化脚本。每次新增迁移后，**必须同步更新** `init-db.sql`，确保其与迁移序列的最终状态一致。

---

## 禁止事项

| 禁止行为 | 原因 |
|----------|------|
| ❌ 手动 `ALTER TABLE` | 破坏可重复部署，无法追溯变更历史 |
| ❌ 跳过迁移执行器直接用 `mysql` 命令 | 绕过迁移记录表 |
| ❌ 修改已部署到生产环境的迁移文件 | 应新增迁移修正 |
| ❌ 删除已有迁移文件 | 即使不用也保留，维持历史完整性 |
| ❌ 在应用代码中执行 DDL | 应用层只做 DML，DDL 走迁移 |

---

## 迁移执行器模板

```js
// scripts/run-migration-NNN-<description>.js
const path = require('path');
const { applySqlFile } = require('./apply-sql-file');

const migrationFile = path.join(__dirname, '..', 'migrations', 'NNN_<name>.sql');

applySqlFile(migrationFile)
  .then(() => { console.log('Migration NNN applied successfully.'); process.exit(0); })
  .catch(err => { console.error('Migration NNN failed:', err); process.exit(1); });
```
