# 移动端 App 开发计划（V1）

项目：Dorm · XMUMDorm-2.0.0-LYZZ  
编写日期：2026-06-01  
目标：**Web 端保留不变**，新增一个**独立移动端 App（iOS/Android）**，与 Web **共用同一后端 API**，逐步覆盖 Web 的全部核心功能。

---

## 0. 背景与边界

### 0.1 已有基础（仓库现状）

- **后端**：Node/Express（`server.js` + `routes/*`），默认端口 `4040`，API 统一以 `/api/*` 提供。
- **Web 前端**：React + Vite + Tailwind（`frontend/`），UI/交互已较完整，含管理员后台。
- **移动端**：已存在 `mobile/`（Expo + React Native）作为移动端工程雏形，已实现部分页面/组件、鉴权存储与 API 请求封装。

### 0.2 本计划的严格约束

- **Web 端继续存在并独立发布**：不因移动端而迁移或替换 Web。
- **移动端 App 与 Web 端共用同一后端**：移动端只调用既有 `/api/*`，必要时允许新增少量“移动端友好”的 API（必须保持向后兼容）。
- **账号体系与权限体系一致**：Token/JWT、管理员/商家/学生角色沿用后端既有实现。

## 1. 技术路线（两条都可行，按目标选择）

> 你的诉求是“另外做一个移动端 App，用同一个后端”。这里给出两条路线，区别在于 **UI 复用程度** 与 **研发成本/体验**。

### 路线 A：React Native（Expo）原生实现（长期推荐）

- **定位**：移动端独立 UI，但视觉规范/组件体系对齐 Web 端设计（“同设计语言”，不强求一份代码跑两端）。
- **优点**：原生体验更好、可深度使用系统能力（推送/相机/文件/分享/手势）。
- **成本**：需要把 Web 页面逐步“移动端化”重做；可复用的是 **设计规范、文案、业务逻辑、API、数据结构**。

### 路线 B：WebView + 壳（最快上线）

- **定位**：App 内嵌 Web UI（可接近 100% 复用现有 Web 页面），App 主要提供原生能力与容器能力。
- **优点**：上线最快、UI 复用最大。
- **成本/风险**：长列表性能、键盘/返回手势、WebView 兼容性与桥接复杂度；后续仍可能需要 RN 重写部分模块。

> 本计划默认按 **路线 A（RN 原生）** 来拆里程碑；若你决定走路线 B，可把“页面开发”替换为“WebView 容器 + 桥接”，其它后端共用、配置、发布流程基本相同。

## 2. 里程碑与排期（V1）

### M0：项目落地与规范化（1–2 天）

- **工程与依赖**
  - 固定移动端技术栈：Expo SDK、导航方案、网络层、状态管理与请求缓存（已使用 React Query）。
  - 统一 TypeScript/ESLint/Prettier（如暂缺）。
- **目录规范**
  - `mobile/src/screens/`：页面
  - `mobile/src/components/`：组件
  - `mobile/src/api/`：API 封装（与 Web 对齐命名与错误模型）
  - `mobile/src/context/`：Auth/Language/Toast 等上下文
- **验收**
  - `mobile` 可在真机/模拟器启动；能连通后端健康接口（如 `/api/users/me` 或任一公开接口）。

### M1：环境配置与网络可用性（1–2 天）

- **任务**
  - 移除“写死局域网 IP”的 API 配置，改为可切换环境：
    - `EXPO_PUBLIC_API_BASE_URL`（开发/测试/生产）
    - EAS build profile 注入（如使用 EAS）
  - 规范接口地址策略：生产必须走 HTTPS 域名，开发走局域网 IP 或本机代理。
  - 统一图片/上传 URL 拼接（与 Web 的 `getUploadUrl()` 语义一致）。
- **验收**
  - 换 Wi-Fi/换电脑 IP 不需要改代码，只改环境变量即可。

### M2：登录态与鉴权体系（2–4 天）

- **任务**
  - 移动端登录/登出、Token 存储（AsyncStorage），与后端 JWT 规则一致。
  - 启动时恢复登录态、拉取 `/me` 刷新用户信息。
  - 统一错误处理（401/403/500），提供友好的“需要登录/无权限/网络异常”页面与重试机制。
- **验收**
  - 登录后可正常访问需要登录的接口；退出后所有受保护页面自动回到登录页。

### M3：导航框架与基础页面骨架（2–5 天）

- **任务**
  - 建立 Tab 结构：树洞 / 食堂 / 信箱 / 广场 / 我的
  - 建立通用页面骨架：安全区、统一 Header、空状态、加载态、错误态、下拉刷新。
  - 统一视觉变量（颜色/圆角/阴影/字体），对齐 Web 端“类 iOS”设计语言。
- **验收**
  - 五大 Tab 可切换，页面状态完整，UI 风格一致。

### M4：核心功能一期（按优先级拆分，2–4 周）

> 目标：先把“高频 + 核心闭环”做出来，再补低频模块与运营后台。

- **树洞（优先级 P0）**
  - 列表/瀑布流、标签筛选、详情、评论、发布、图片上传、点赞/收藏（若后端支持）。
- **食堂（P0）**
  - 区域列表、店铺/菜品列表、菜品详情、点评列表/发布点评（含图片）。
- **信箱（P1）**
  - 通知/公告列表、已读/未读、详情。
- **我的（P0）**
  - 用户信息、头像、设置、语言切换、入口集合（课表/待办/日记等逐步补齐）。

**验收（一期）**
  - 能完成“登录 → 浏览内容 → 互动（评论/发布）→ 查看个人信息”的主链路。

### M5：模块扩展（二期，2–6 周）

- **广场/社团/二手/跑腿/手册/课程点评**等按业务优先级逐一落地。
- 对齐 Web 端功能：
  - 列表分页/搜索
  - 发布与编辑
  - 管理与删除（按角色）

### M6：运营与后台（可选，1–3 周）

- 若管理员确实需要在手机上做运营：
  - 优先实现“轻量后台”：举报处理、公告发布、敏感词开关、日志查看。
  - 复杂表格/批量操作可留在 Web 管理后台。

## 3. 开发任务拆解（工程化）

### 3.1 API 兼容与契约

- **契约原则**
  - 移动端与 Web 端共用同一套 `/api/*`，返回结构（`{status, message, data}`）一致。
  - 移动端不依赖“Web 专用字段”，如有差异应在后端补齐或在移动端适配。
- **新增 API 的规则**
  - 必须保持向后兼容；新增字段不破坏旧端。
  - 高风险改动需要补测试（至少路由级 Jest）。

### 3.2 组件与设计系统

- 建议建立移动端 `ui/` 组件集合（Button/Input/Card/EmptyState/Toast/Modal），并形成统一主题：
  - 色板、字号、圆角、间距、阴影、分割线、状态色（成功/警告/错误）
- 与 Web 对齐方式：
  - 对齐“视觉规范”，而非强行共享 CSS/Tailwind。

### 3.3 质量与测试

- **最低要求**
  - 关键 API 失败兜底（空态/错误态/重试）
  - 关键流程回归清单（见第 6 节）
- **建议**
  - 后端保持路由测试覆盖（已有 `__tests__` 体系）。
  - 移动端引入最小化的 e2e/冒烟脚本（可后续）。

## 4. 原生能力规划（按需启用）

- **推送通知**：expo-notifications，后端已有 push 相关 API（需对齐 token 上报协议）。
- **图片上传**：expo-image-picker + 后端 `/uploads`（需处理多图、压缩、失败重试）。
- **深链**：通知点击跳转到指定内容详情。
- **缓存与离线**：React Query 缓存 + 本地持久化（可选）。

---

## 5. 风险与对策

- **环境配置不稳定（局域网 IP 变动）**
  - 对策：强制使用环境变量 + profile；开发写入 README；禁止硬编码。
- **接口差异/字段缺失导致移动端开发受阻**
  - 对策：以“后端契约”为中心补齐字段；必要时新增聚合接口，但保持兼容。
- **性能风险（长列表、图片瀑布流）**
  - 对策：分页 + 图片懒加载 + 压缩；必要时上性能监控与分批渲染策略。

## 6. 验收清单（每次发布前）

- **账号**
  - 注册/登录/退出
  - Token 过期重登
- **核心页面**
  - 树洞列表/详情/发帖/评论/上传图
  - 食堂区域/店铺/菜品/点评/发布点评
  - 信箱通知列表与详情
  - 我的页面信息显示与设置
- **网络与异常**
  - 断网/超时/后端 500 的提示与重试
- **权限**
  - 非管理员不可访问管理员接口
  - 角色行为与 Web 一致

## 7. 建议的交付节奏（可用于你自己的排期）

- 第 1 周：M0–M2 完成（工程、配置、登录）
- 第 2–3 周：M3 + 树洞（P0）+ 食堂（P0）主链路
- 第 4 周：信箱（P1）+ 我的（P0）完善 + 发布准备
- 后续：按模块逐步补齐广场/社团/二手/跑腿/手册/课程点评；再考虑轻量管理员能力
