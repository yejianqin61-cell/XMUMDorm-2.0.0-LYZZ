# 回滚后现状评估与合理开发顺序

> 回滚到稳定版本后整理。目标：**有头绪、有顺序**地开发，避免再次「一股脑」导致漏洞百出。

---

## 一、现状评估

### 1.1 总体结论

| 层级   | 状态说明 |
|--------|----------|
| **后端** | 已较完整：认证、帖子、食堂、通知、用户、排行榜（含每周重置）均有路由与逻辑，数据库有 init-db + 三份迁移（帖子 / 食堂 / 排行榜）。 |
| **前端** | 仅**登录、注册**接真实 API；其余页面（树洞、帖子详情/发帖、食堂、信箱、个人空间等）仍用**假数据**或占位，未与后端打通。 |

也就是说：**后端可复用，前端要「按模块、按顺序」逐步接 API，而不是一次性大改。**

---

### 1.2 后端现状（简要）

| 模块     | 路由/入口              | 说明 |
|----------|------------------------|------|
| 认证     | `routes/auth.js`       | 注册、登录、JWT，已用 |
| 帖子     | `routes/posts.js`      | 发帖、列表、详情、点赞、评论、删除、隐藏、图片上传 |
| 食堂     | `routes/canteen.js`    | 区域、店铺、分类、商品、评论（含评级/图片）、逻辑删除 |
| 通知     | `routes/notifications.js` | 通知列表等 |
| 用户     | `routes/users.js`      | 用户信息、个人资料等 |
| 排行榜   | `services/rankingStats` + server 定时 | 每周一东八区重置等 |

数据库：`init-db.sql`（用户 + 帖子相关表）+ `migrations/001_posts_*`、`002_canteen_*`、`003_ranking_*`。  
若从未跑过迁移，需按顺序执行一次。

| 页面/功能       | 当前状态           | 与后端关系     |
|-----------------|--------------------|----------------|
| 登录 / 注册     | 正常使用           | ✅ 已接真实 API |

## 二、合理的开发顺序（推荐）

原则：**一个垂直功能做完、自测通过、再做下一个；每步小提交，避免一次改太多。**

---

### 阶段 0：环境与数据（若未做过）

- [ ] 确认本地/测试环境数据库已建库并执行：`init-db.sql` → `001_posts_*` → `002_canteen_*` → `003_ranking_*`。
- [ ] 后端 `npm run dev`、前端 `npm run dev` 能正常跑，登录/注册能成功。

---

### 阶段 1：帖子系统前端（树洞 + 发帖 + 详情）

**目标**：用户能看到真实帖子列表、点进去看详情、能发新帖。只做「读列表 + 读详情 + 发帖」这一条线，不顺便改食堂或其它。

1. **1.1 封装帖子 API**  
   在 `frontend/src/api/` 下新增 `posts.js`（或统一 `api.js`），封装：
   - `GET /api/posts`（列表，分页参数）
   - `GET /api/posts/:id`（详情）
   - `POST /api/posts`（发帖，含图片时用 FormData）
   请求头里带 `Authorization: Bearer <token>`（从 AuthContext 取）。

2. **1.2 树洞页接真实列表**  
   - 在 `TreeHole.jsx` 里用 `useEffect` 调 `GET /api/posts`，替换 `MOCK_POSTS`。
   - 处理 loading、空列表、错误提示；保留现有布局与样式，只改数据源。

3. **1.3 帖子详情页接真实详情与评论**  
   - `PostDetail.jsx` 根据路由 `id` 调 `GET /api/posts/:id` 和评论接口（若有 `GET /api/posts/:id/comments`）。
   - 替换 mock 数据和 mock 评论，保留 UI 结构。

4. **1.4 发帖页接真实发帖**  
   - `PostNew.jsx` 表单提交时调 `POST /api/posts`（文字 + 图片），成功后可跳转回树洞或该帖详情。
   - 校验、错误提示、上传进度可按需简单做。

5. **1.5 小优化（可选，放在本阶段末尾）**  
   - 点赞：在详情或卡片上调用点赞 API，并刷新列表/详情。
   - 删除帖子：仅本人或管理员，调用删除接口后刷新或跳转。

**提交建议**：每完成 1.1、1.2、1.3、1.4 各一次小提交，例如「api: 帖子列表/详情/发帖」、「feat: 树洞接真实帖子列表」。

---

### 阶段 2：食堂系统前端（Eat 页）

**目标**：学生能按区域看商家、点进商家看商品、点进商品看详情与评论；商家相关（创建店铺、商品）可再拆子阶段。先做「浏览」这条线。

1. **2.1 封装食堂 API**  
   - 在 `frontend/src/api/` 下保留/整理 `canteen.js`，封装：
     - `GET /api/canteen/regions`
     - `GET /api/canteen/regions/:regionId/shops`
     - `GET /api/canteen/shops/:shopId`（店铺详情含分类与商品）
     - `GET /api/canteen/products/:id`（商品详情）
     - 评论：`GET/POST /api/canteen/products/:id/comments`（你已有部分可复用）

2. **2.2 Eat 页：区域列表 + 按区域看店铺**  
   - 先只做：顶部或左侧区域 Tab/列表，选区域后请求该区域下的店铺列表并展示。
   - 不先做店铺详情、商品详情，避免一次堆太多。

3. **2.3 店铺页（可新页面或 Eat 内子视图）**  
   - 点击某店铺 → 请求该店铺详情（分类 + 商品），按分类展示商品列表。

4. **2.4 商品详情页**  
   - 点击某商品 → 新页面或抽屉，请求商品详情 + 评论列表；可先只读评论，发评论可放在 2.5。

5. **2.5 发评论、评级**  
   - 在商品详情页接「发表评论」接口（含必填评级、文字、可选图片），提交后刷新评论列表。

**提交建议**：2.1 一次、2.2～2.5 各一次或按你习惯拆成更小步。

---

### 阶段 3：个人相关（我的帖子、资料、信箱）

**目标**：我的帖子用真实接口；个人资料编辑、信箱用真实接口替换 mock。

1. **3.1 我的帖子**  
   - 若有 `GET /api/posts?authorId=xxx` 或用户维度的帖子接口，在 `MyPosts.jsx` 里调用并替换 `getPostsByAuthorId` 等 mock。

2. **3.2 个人资料编辑**  
   - `ProfileEdit.jsx` 接「获取当前用户信息」+「更新用户信息」接口（如 `GET/PATCH /api/users/me` 或类似），替换 mock。

3. **3.3 信箱（通知）**  
   - `Mailbox.jsx` 接通知列表接口（如 `GET /api/notifications`），替换 `MOCK_NOTIFICATIONS`；点击跳转逻辑与后端返回结构对齐。

**提交建议**：一个功能一个提交。

---

### 阶段 4：商家端（可选、可再拆）

若需要商家在网页端创建店铺、分类、商品：

- 先做「创建店铺」（选区域 + 店铺名），再做「分类管理」「商品管理」（列表、新建、编辑、删除）。
- 每块单独分支、小步提交，避免和阶段 2 学生端混在一起改。

---

### 阶段 5：排行榜与其它

- 若设计文档里有排行榜前端（榜单页、周榜等），在帖子与食堂都接好后，再单独做一页接排行榜 API。
- 其它小功能（关于我们、设置等）按优先级插在以上阶段之间或之后。

## 三、顺序总结（一张表）

| 顺序 | 内容                     | 产出 |
|------|--------------------------|------|
| 0    | 环境与数据库迁移         | 能登录、后端接口正常 |
| 1    | 帖子前端：API 封装 → 树洞 → 详情 → 发帖 | 树洞全流程真实数据 |
| 2    | 食堂前端：API 封装 → 区域/店铺列表 → 店铺详情 → 商品详情 → 评论 | Eat 页可浏览、可评 |

| 4    | 商家端（可选）           | 商家可管理店铺与商品 |
| 5    | 排行榜及其它             | 按需 |

## 四、避免再次「乱掉」的几点建议

1. **一次只做一个小目标**：例如「本周只做树洞接真实列表」，不做食堂、不改信箱。
2. **先接再美化**：先让接口跑通、数据对，再调样式与交互。
3. **小步提交**：每完成一个小功能就 commit，方便回滚和排查。
4. **有疑问先对设计**：食堂逻辑以《食堂系统设计说明_待确认.md》为准；帖子以《帖子系统需求与设计说明_2.0.0.md》为准，避免临时加需求导致前后端不一致。
5. **需要大改时开分支**：例如「食堂前端大改」在 `feature/eat-page` 上做，合并前再整体测一遍。

按这个顺序做，前端会一步步和现有后端对齐，且每一步都可控、可回滚。若你告诉我当前想先做「树洞」还是「食堂」，我可以按该阶段拆成更细的 TODO（仍用 MD 或列表形式）。
