# 食堂 Web 信息流治理规格

## 问题

当前食堂首页同时呈现搜索、轮播、分区、榜单、随机推荐、食评流和内部侧栏。页面模块均以独立卡片呈现，且和全站三列主壳叠加后形成重复层级。分区商家、商家菜单、菜品详情和点评页也分别使用不同的卡片与信息层级，学生难以快速完成“找食堂、找店和菜、判断值不值得吃”。

## 方案

保留全站左侧导航、中间内容列、右侧辅助信息区。食堂模块不再在中间内容列内建立第二个侧栏。

`/eat` 收敛为连续信息流：搜索和食堂分区在首屏；随后是“大家都在吃”的菜品图卡；最后是紧凑的食堂讨论流。轮播仅在存在真实运营内容时显示；随机推荐收进菜品推荐标题旁的图标操作。

分区页负责比较商家；商家页保留分类导航与菜品浏览；菜品详情按图片、决策信息、点评流的顺序阅读。所有改动复用已有查询、接口、路由和 token，不增加后端 API。

## 用户故事

1. 作为学生，我想在食堂首页先搜索或选择食堂分区，以便快速开始找吃的。
2. 作为学生，我想直接看到受欢迎的菜品，以便不必逐个进入食堂比较。
3. 作为学生，我想通过菜品图片、名称、商家、价格和评分快速比较，以便判断是否值得吃。
4. 作为学生，我想点击一个分区后连续浏览该分区商家，以便选择去哪里吃。
5. 作为学生，我想在商家菜单中通过分类定位菜品，以便快速翻到目标类别。
6. 作为学生，我想在菜品详情中先看到图片、价格、评分和商家，再读点评，以便快速决策。
7. 作为学生，我想阅读近期食评帖子，以便获得真实同学反馈。
8. 作为学生，我想从食评流直接进入帖子详情或发布食评，以便继续参与讨论。
9. 作为手机用户，我想在窄屏上仍能完成搜索、选分区、看菜和写点评，以便不因桌面布局受阻。
10. 作为商家或管理员，我想保留现有管理、榜单、地图和发布入口，以便治理只改变用户阅读体验而不丢失功能。

## 实施决定

- 首页删除说明性 hero、统计、默认轮播和内部快捷入口侧栏。
- 首页首屏仅保留搜索与食堂分区；默认信息流仅保留菜品推荐和食堂讨论两段。
- 随机推荐复用现有随机菜品能力，作为菜品推荐标题旁的图标操作，不单独占用大区块。
- 轮播复用现有运营数据，仅在存在可展示内容时渲染。
- 分区商家列表使用连续列表和轻分割，展示已有且可靠的商家名称、分区、评分、营业信息与简介字段。
- 菜品卡改为双列图片卡，图片是主要扫描锚点；菜品详情按决策信息优先排序，不重复渲染快照和操作卡。
- 点评使用连续阅读流，发布入口只在点评区域内出现，不能遮挡菜品图片。
- 全站右侧辅助区仍由主壳负责；食堂页面内部不再创建额外右栏。

## 测试决定

- 以现有路由和 API 调用为最高测试边界，验证用户能完整走通 `/eat -> 分区 -> 商家 -> 菜品 -> 点评`。
- 对首页验证：无运营轮播时不留下空白；搜索、分区、菜品推荐和食堂讨论均可进入下级路径。
- 对列表与详情验证：已有价格、评分、商家、图片和点评正常显示；无数据时提供一个直接动作。
- 每个任务完成后运行 `frontend` 生产构建，并在桌面、平板和手机宽度人工检查内容不会被内部侧栏或固定操作遮挡。

## 范围外

- 不新增收藏、下单、配送、距离、价格等级、智能推荐、营业状态解析或新的后端接口。
- 不改商家后台和管理员运营流程，除非现有用户页面复用组件时必须做兼容调整。
- 不删除地图、排行榜、搜索和发布路由。
