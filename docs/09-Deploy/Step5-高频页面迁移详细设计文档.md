# Step5-高频页面迁移详细设计文档

## 1. 文档目标

本文用于承接已经完成的 Step 1 到 Step 4，正式细化 Step 5：

- 在不影响 `frontend-app/` 的前提下，开始迁移 Web 端高频页面
- 让页面从“已进入主壳”继续推进到“真正具备桌面 Web 阅读与操作节奏”
- 把后续页面改造拆成可连续执行、可独立提交、可逐页验收的任务

Step 5 的重点不再是：

- 继续搭壳
- 继续抽基础组件
- 继续抽模板骨架

而是：

- 把高频页面挂进现有壳与模板体系
- 收口移动端遗留的页面级布局假设
- 建立桌面端信息密度、辅助栏、区块节奏和交互落点

## 2. 当前阶段背景

当前仓库已经具备：

- Step 1 Token 层
- Step 2 基础 UI 组件层
- Step 3 页面模板层
- Step 4 Web 主壳与代表页接入验证

这意味着当前的页面改造已经不需要再从零决定：

- 页面放在哪个主容器里
- 页面应该用哪类模板
- 页面该消费哪些 token 与基础件

Step 5 的工作，应该变成：

```text
选高频页
-> 判断它归属哪一类模板
-> 把移动端页面级外壳假设降下来
-> 让它在主壳里形成桌面阅读/操作节奏
-> 验证不回归
```

## 3. 当前高频页面现状判断

### 3.1 已经具备主壳承载关系，但仍偏移动页面的页面

典型代表：

- `frontend/src/pages/TreeHole.jsx`
- `frontend/src/pages/CanteenHome.jsx`
- `frontend/src/pages/PostSearch.jsx`
- `frontend/src/pages/PostTagFeed.jsx`
- `frontend/src/pages/Mailbox.jsx`

这类页面的共同问题通常不是“没进壳”，而是：

- 页面仍默认全屏沉浸式移动节奏
- 页面内部仍带强烈的手机端滚动、留白、浮层、双列假设
- 壳下内容区与页面自身布局结构容易重复叠加

### 3.2 已经验证过模板，但还没完成桌面化扩散验证的页面

典型代表：

- `frontend/src/pages/PostDetail.jsx`
- `frontend/src/pages/FoodDetail.jsx`
- `frontend/src/pages/ProfileEdit.jsx`
- `frontend/src/pages/MyPosts.jsx`
- `frontend/src/pages/MyReviews.jsx`

这类页面的问题通常不是“模板不可用”，而是：

- 模板虽然接上了，但桌面壳下的内容宽度、区块关系、旁侧信息还不够稳
- 页面 CSS 里仍保留移动端结构假设
- 同类页面之间的节奏还没真正统一

### 3.3 最值得优先推进的高频路径

结合现有文档与页面重要度，Step 5 首批建议优先推进：

1. 广场内容流
   - `TreeHole`
   - `PostSearch`
   - `PostTagFeed`
2. 食堂核心入口
   - `CanteenHome`
3. 高频详情链路
   - `FoodDetail`
   - `MerchantFoodDetail`
4. 我的次级高频入口
   - `Mailbox`
   - `MyPosts`
   - `MyReviews`

## 4. Step 5 的核心目标

Step 5 只解决“高频页面桌面化迁移”，核心目标有 5 个：

1. 让高频页面在主壳下看起来像真正的网页，而不是壳里的手机页。
2. 复用 Step 3 模板，不回头重造页面结构。
3. 降低页面级 CSS 对全屏移动容器、固定底栏、沉浸式滚动的假设。
4. 用少量代表页验证同类页面的可批量迁移性。
5. 为 Step 6 模块级批量治理建立一套已验证的迁移路径。

## 5. 本轮边界

### 5.1 本轮要做的事

- 拆高频页面迁移顺序
- 明确每一类页面该怎么进入桌面节奏
- 先选代表页验证，再批量扩散
- 对接已有主壳与模板，不重复造新壳或新模板

### 5.2 本轮不做的事

- 不重做 `frontend-app/`
- 不重构共享 API 层
- 不大规模改动业务请求逻辑
- 不在一个 task 里同时治理多个不相干模块
- 不把“页面桌面化”顺手扩大为“全模块彻底重写”

## 6. Step 5 与前置步骤的关系

Step 5 必须明确依赖前面四层，而不是绕过它们：

```text
Token
-> Base UI
-> Page Template
-> Site Shell
-> High-frequency Page Migration
```

也就是说：

- Step 1 提供统一视觉变量
- Step 2 提供统一基础控件
- Step 3 提供页内结构模板
- Step 4 提供站点级承载壳
- Step 5 才开始逐页把真实高频页面迁进去

## 7. 页面分类与迁移策略

## 7.1 内容流页

代表页面：

- `TreeHole`
- `PostSearch`
- `PostTagFeed`
- `SquareCampusFeed`

迁移重点：

- 让主列成为稳定阅读主轴
- 让筛选、标签、搜索从移动顶部条升级为桌面信息工具区
- 降低“全屏沉浸瀑布流”对整体页面结构的支配
- 让辅助推荐内容与内容流节奏形成稳定关系

不建议本轮做的事：

- 不重写帖子接口
- 不重做无限滚动机制
- 不把瀑布流算法优化和页面桌面化混成同一个 task

## 7.2 门户首页页

代表页面：

- `CanteenHome`
- 后续可类推到模块首页型页面

迁移重点：

- 从移动端模块堆叠页转成桌面门户首页
- 区分“导航入口区、主推荐区、次级工具区、榜单或说明区”
- 明确桌面端首屏信息层次

不建议本轮做的事：

- 不顺手重写所有食堂二级页
- 不重写商家/菜品 API

## 7.3 详情页

代表页面：

- `FoodDetail`
- `MerchantFoodDetail`
- 后续 `MarketplaceDetail`、`ClubProfile`

迁移重点：

- 正文主轴受控
- 操作区、元信息区、补充说明区分层
- 壳下详情页不要再保留移动页那种全宽单列挤压感

## 7.4 我的次级高频页

代表页面：

- `Mailbox`
- `MyPosts`
- `MyReviews`

迁移重点：

- 让“我的”链路不只首页工作台像桌面页
- 保持与 `MyZone` 的节奏连续性
- 统一个人页下的列表页、消息页、工具页视觉落点

## 8. 首批代表页选择

### 8.1 TreeHole 作为 Step 5 首个代表页

优先选择 `TreeHole` 的原因：

- 高频
- 当前视觉上仍明显偏移动沉浸式内容流
- 已经在主壳导航体系中有稳定位置
- 很适合验证“主壳下高密度内容页”的桌面化策略

它最适合验证的不是业务逻辑，而是：

- 内容流在桌面主列中的节奏
- 工具栏和标签栏的桌面落点
- 推荐块与帖子流的主次关系
- 页面滚动与壳容器的边界

### 8.2 CanteenHome 作为第二类代表页

优先选择 `CanteenHome` 的原因：

- 是模块首页
- 使用频率高
- 最能体现“网页首页布局”与“手机模块首页”的区别

### 8.3 Mailbox 作为“我的次级高频页”代表页

优先选择 `Mailbox` 的原因：

- 高频入口明显
- 与工作台链路天然连续
- 容易验证“我的模块”在主壳下是否形成一致节奏

## 9. 迁移实施原则

### 9.1 一次只动一个页面或一组强相关页面

允许：

- `TreeHole` 单页 task
- `PostSearch + PostTagFeed` 成对 task
- `FoodDetail + MerchantFoodDetail` 同类详情 task

不建议：

- 一个 task 同时改 TreeHole、Mailbox、CanteenHome

### 9.2 先收结构，再收视觉

每个页面迁移建议固定顺序：

1. 先确认壳下落位
2. 先调整页面最外层结构
3. 再收主列与旁侧区关系
4. 再补视觉统一与细节动效

### 9.3 优先降移动端假设，不优先发明新结构

后续代码里优先寻找并降掉这些常见假设：

- 全屏容器写死 padding 与 safe-area
- 页面默认依赖底部 tab 占位
- 页面默认自己做整页 header、整页滚动控制
- 页面默认没有桌面主列/旁侧区概念

### 9.4 页面业务逻辑尽量不动

Step 5 的默认原则是：

- 优先动 JSX 结构
- 优先动页面 CSS
- 优先动少量与布局强耦合的展示组件

不到必要时，不主动重写：

- query key
- mutation 流程
- 数据映射逻辑

## 10. Step 5 建议任务序列

建议 Step 5 首批按下面顺序推进：

1. `TreeHole`
2. `PostSearch + PostTagFeed`
3. `CanteenHome`
4. `FoodDetail + MerchantFoodDetail`
5. `Mailbox`
6. `MyPosts + MyReviews`

这个顺序的好处是：

- 先打最能看出桌面化成果的内容流页
- 再打门户首页
- 再打详情页
- 最后收我的次级高频链路

## 11. 验收标准

Step 5 文档阶段验收至少应满足：

1. 已明确首批高频页面名单与优先级。
2. 已明确每类页面的迁移方式与边界。
3. 已明确首批代表页为何这样选。
4. 已明确不应混做的内容。
5. 已能继续拆成正式 task 文档。

进入代码实施后，每个页面 task 还应补充：

1. 页面在桌面主壳下结构更稳定。
2. 页面不再强依赖移动端整屏布局假设。
3. 页面主要业务行为不回归。
4. `npm run build:web` 通过。

## 12. 如果做错了，会出现什么表征

### 12.1 页面只是进了壳，但仍像手机页

表征：

- 页面内容过窄或过满
- 页面整体节奏仍像纵向手机卡片堆叠
- 桌面端没有清晰主列、区块或辅助区

### 12.2 页面自己继续抢壳职责

表征：

- 页面自己又定义整页 header
- 页面自己继续控制整屏滚动逻辑
- 页面 padding / width 与壳容器相互覆盖

### 12.3 一个 task 动太多层

表征：

- 同时改业务逻辑、接口层、模板层、主壳层
- 问题一旦出现，无法定位是页面问题还是系统问题

### 12.4 同类页面仍没有形成批量迁移能力

表征：

- 做完 `TreeHole` 后，`PostSearch` 仍然完全不能复用思路
- 做完 `FoodDetail` 后，`MerchantFoodDetail` 还是得重做一遍

## 13. 需要你把关的点

正式进入 Step 5 执行前，最值得继续把关的是：

1. 是否确认 `TreeHole` 作为 Step 5 首个执行页。
2. 是否接受 `PostSearch` 与 `PostTagFeed` 作为同一组列表/搜索页联合治理。
3. 是否接受 `CanteenHome` 作为门户首页型代表页。
4. 是否接受 `Mailbox` 作为“我的次级高频页”代表页。
5. 是否继续坚持“默认不动 API 层，只收结构与样式”。

## 14. 结论

Step 5 的关键不是“继续随便挑几页改”，而是正式进入高频页面桌面化迁移阶段。

只要这一阶段的代表页和拆解策略先定稳，后面就可以按：

`代表页验证 -> 同类页扩散 -> 模块级治理`

这个顺序持续推进，而不是重新回到“一页一页边改边想”的状态。
