# Web 前端视觉语言统一设计方案

## 1. 方案目标

本方案用于统一 `frontend/` 的 Web 前端视觉语言，确保后续桌面化改版时：

- 全站风格一致
- 页面之间不再各自为战
- 视觉、动效、布局、组件都能复用
- 保持浅色调、活泼、校园风的品牌气质

本方案默认服务对象是：

- 产品主题：大学校园生活平台
- 核心用户：在校学生、社团成员、二手交易用户、内容发布者、管理员
- Web 首页与主站的单一任务：让用户像逛“校园公共空间”一样自然浏览、参与、发布和管理内容

## 2. 视觉总方向

### 2.1 风格关键词

全站统一风格关键词定为：

- 浅色调
- 明亮
- 轻盈
- 青春
- 活泼
- 有校园公告栏与社团海报感
- 带一点手作感，但不幼稚

### 2.2 视觉核心判断

这套 Web 不是走：

- 科技 SaaS 冷淡风
- 高级奢侈感极简风
- 黑色赛博风
- 通用后台模板风

而是走“校园公共空间”视觉路线：

- 像阳光下的教学楼连廊、公告栏、社团展板、食堂指示牌
- 信息很多，但不压抑
- 色彩轻快，但不过度花哨
- 要让用户感到“亲近、可信、愿意停留”

## 3. 品牌视觉主张

### 3.1 设计主张

统一主张：

> 像校园里一块被认真维护的公共信息墙，明亮、友好、有活力，信息丰富但始终清爽。

### 3.2 记忆点

全站最重要的视觉记忆点建议定为：

- “纸张感卡片 + 晨光色背景 + 彩色模块标签”

这会让页面既不像普通管理台，也不像千篇一律的内容社区。

### 3.3 审美风险点与取舍

本方案愿意承担的一个设计风险是：

- 引入轻微“校园纸面感”和“公告栏感”的卡片系统

理由：

- 这比普通白卡片更有辨识度
- 很贴合校园内容社区的主题
- 只要控制好留白、阴影和色彩比例，就不会显得廉价

- 不同模块可以有自己的辅助色，但必须在统一色板内取值
- 不允许每个页面自己定义一套主色

## 5. 模块色语义

为了让用户一眼识别内容模块，建议给核心模块定义固定辅助色：

| 模块 | 辅助色 | 说明 |
| --- | --- | --- |
| 广场 / 首页 | `#4DA7FF` | 清爽、开放、信息流入口 |
| 树洞 / 帖子 | `#7C9BFF` | 轻社交、表达欲 |
| 食堂 | `#FFB86C` | 温暖、食物感 |
| 社团 | `#FF8FB1` | 活力、活动感 |
| 二手 | `#73D9B3` | 轻交易、可信赖 |
| 新生指南 | `#8EC5FF` | 友好、帮助感 |
| 跑腿 | `#FFD66B` | 快捷、行动感 |
| 我的 | `#A7B4FF` | 个人空间、柔和稳定 |
| 后台 | `#5F7A99` | 更克制、更专业 |

要求：

- 模块色只做识别，不做大面积铺底
- 页面仍共享统一背景和卡片体系
- 模块色主要用于标题标签、图标、按钮强调和空态插画点缀

## 6. 字体与排版系统

### 6.1 字体角色

建议采用三层字体角色：

- 展示字体：用于首页大标题、活动标题、重点 Banner
- 正文字体：用于正文、列表、表单、卡片内容
- 工具字体：用于数字、标签、状态、统计信息

### 6.2 字体建议

在不强依赖商用字体的前提下，建议方向如下：

- 展示字体：
  - 优先考虑有一点书卷气和亲和感的中文标题字体
  - 不要太硬朗，也不要过于手写
- 正文字体：
  - 现代、清晰、长时间阅读不累
- 工具字体：
  - 更规整、更适合数字和小标签

如果后续需要具体落地建议，方向可以是：

- 中文标题：偏人文、圆润、轻书卷气
- 中文正文：偏现代无衬线
- 英文与数字：偏几何无衬线

### 6.3 字号层级

| 层级 | 建议字号 | 用途 |
| --- | --- | --- |
| Display XL | `48-56px` | 首页视觉主标题 |
| Display L | `36-40px` | 模块首页大标题 |
| H1 | `28-32px` | 页面主标题 |
| H2 | `22-24px` | 模块标题 |
| H3 | `18-20px` | 卡片标题、二级标题 |
| Body L | `16px` | 主正文 |
| Body M | `14px` | 辅助正文 |
| Caption | `12px` | 时间、meta、标签 |

### 6.4 排版原则

- 标题要有“公告栏标题感”，但不要厚重
- 正文行高保持舒适，建议 `1.6 - 1.75`
- 页面标题与模块标题保持固定层级关系
- 不允许某页很松、某页很挤

## 7. 背景与氛围系统

### 7.1 全站背景策略

全站背景不建议纯白到底，建议使用轻氛围背景：

- 主背景使用非常淡的天蓝灰白
- 局部可叠加柔和渐变
- 少量使用抽象几何光斑或纸张纹理感

### 7.2 背景表达原则

- 背景必须轻，不抢内容
- 不能使用重渐变和强对比噪点
- 重点是“空气感”和“晨光感”

### 7.3 模块背景建议

- 首页 / 广场：轻天光渐变
- 食堂：奶油橙调光斑
- 社团：粉蓝彩纸感点缀
- 指南：更清爽、更安静
- 后台：去装饰化，保持专业

## 8. 卡片系统

### 8.1 卡片是主角

这套站点最重要的基础组件不是按钮，而是卡片。

原因：

- 当前业务本质上是信息流和内容块
- 校园风最适合用“内容展板 / 纸张卡片”来统一承载

### 8.2 卡片统一规则

所有主卡片统一遵循：

- 白底或极浅色底
- 细边框
- 柔和阴影
- 中等圆角
- 顶部或角标带模块色标签

建议参数：

- 圆角：`18px - 24px`
- 边框：`1px solid #D9E4F0`
- 阴影：轻、短、扩散柔和
- 内边距：`20px - 28px`

### 8.3 卡片层级

| 层级 | 用途 |
| --- | --- |
| Hero Card | 首页主视觉、推荐模块 |
| Content Card | 帖子、食物、活动、指南条目 |
| Utility Card | 统计、快捷入口、状态提示 |
| Soft Card | 空态、帮助信息、次级补充内容 |

## 9. 按钮与控件风格

### 9.1 按钮风格

按钮统一要求：

- 轮廓圆润
- 不走强科技感玻璃按钮
- 重点是轻快、清晰、友好

按钮角色建议：

| 类型 | 风格 |
| --- | --- |
| Primary | 品牌蓝实心按钮 |
| Secondary | 白底描边按钮 |
| Tertiary | 文字按钮 |
| Accent | 模块色按钮，用于特定场景 |
| Danger | 浅红警示按钮 |

### 9.2 输入框风格

输入框统一要求：

- 高度适中
- 圆角统一
- 聚焦态出现柔和蓝色外发光
- 错误态只在边框和提示上表达，不整块变红

### 9.3 筛选器与标签

筛选器和标签建议用“校园贴纸感”语言：

- 圆角胶囊
- 颜色轻
- 激活态清楚
- 不要做成企业后台小方块按钮

## 10. 图标、插图与视觉元素

### 10.1 图标风格

图标统一方向：

- 线性图标为主
- 略圆角
- 不要过细
- 不要过锐利

### 10.2 插图风格

如果后续补空态和引导图，统一方向建议：

- 扁平轻插画
- 简单校园元素
  - 教学楼
  - 公告栏
  - 食堂托盘
  - 书本
  - 社团海报
  - 背包
- 配色服从主色板

### 10.3 装饰元素

允许少量使用：

- 彩色圆点
- 轻弧线
- 便签感小标签
- 柔和高光块

但要求克制，不能每屏都装饰。

## 11. 动效系统

### 11.1 动效原则

动效关键词：

- 轻
- 柔和
- 有呼吸感
- 有校园活力
- 不晕、不抢戏

### 11.2 全站动效策略

建议只保留三类核心动效：

1. 页面进入
2. 卡片出现
3. 按钮 / 标签 / Tab 的反馈

不要把页面做成动画游乐场。

### 11.3 页面进入动效

统一建议：

- 轻微上移 + 淡入
- 时长 `240ms - 360ms`
- 缓动使用柔和 ease-out

适用于：

- 模块首页
- 列表切换
- 详情页进入

### 11.4 卡片动效

统一建议：

- 首屏卡片分组式 stagger reveal
- 单张 hover 轻抬升 `2px - 6px`
- 阴影略增强

适用于：

- 帖子卡片
- 食物卡片
- 社团活动卡片
- 推荐卡片

### 11.5 操作反馈动效

统一建议：

- 按钮点击有短促压感
- 标签切换有背景滑动或淡变
- Toast 从顶部或右上柔和滑入

### 11.6 禁止项

- 禁止大幅弹跳
- 禁止频繁缩放
- 禁止重度视差
- 禁止复杂 3D 翻转

### 11.7 无障碍要求

- 必须支持 reduced motion
- 动效不能成为功能理解前提

## 12. 页面布局视觉规则

### 12.1 首页 / 广场

首页要像“校园总公告墙”：

- 上方有视觉主标题区
- 中间有主内容流
- 两侧有模块入口和推荐信息
- 信息多，但层次清楚

### 12.2 列表页

列表页视觉要求：

- 先看到模块身份
- 再看到筛选和排序
- 再进入内容流
- 不要一打开就是一片没有边界的白内容

### 12.3 详情页

详情页要像“公告详情页 / 社团海报详情页 / 食品卡详情页”：

- 主体内容稳定居中
- 侧边栏只做补充
- 强化阅读感与信息秩序

### 12.4 表单页

表单页不要像后台配置页面，也不要像手机单列长表单。

统一要求：

- 结构块清晰
- 标题和说明温和
- 操作按钮始终明确
- 可加入轻提示卡帮助用户完成发布

### 12.5 个人中心

个人中心要从“手机我的页”改成“校园工作台”：

- 头像摘要区
- 最近任务 / 消息 / 收藏 / 发布入口
- 日程、待办、邮箱等工具模块并列展示

## 13. 组件统一规则

### 13.0 复用优先原则

组件治理的第一原则不是“先做新组件”，而是“先判断能不能复用”。

后续所有 Web 改版开发都必须遵守：

- 先查现有组件，再决定是否新增
- 先抽象通用结构，再做页面个性化包装
- 优先改造旧组件，不优先平行再造一个新组件
- 能通过 variant 解决的，不新增同类组件
- 能通过组合解决的，不新增巨型万能组件

禁止出现以下情况：

- `Button`、`PrimaryButton`、`MainButton` 同时并存且职责重叠
- `PostCard`、`SquarePostCard`、`FeedPostCard` 只是样式略不同却各写一套
- 不同页面各自维护自己的 `PageHeader`、`EmptyState`、`FilterBar`

判断是否应该新建组件时，统一按下面标准：

1. 是否已经有 70% 以上能力相近的现有组件
2. 是否只是模块皮肤差异，而不是结构差异
3. 是否可以通过 `variant`、`size`、`tone`、`slot` 解决
4. 是否未来至少会被 2 个以上页面复用

如果以上问题大多回答为“是”，就不应该新建重复组件。

### 13.1 必须统一的基础组件

后续所有页面必须优先统一以下组件族：

- `Button`
- `Input`
- `Textarea`
- `Select`
- `Tag`
- `Badge`
- `Card`
- `Tabs`
- `Dropdown`
- `Modal`
- `Toast`
- `EmptyState`
- `ErrorState`
- `PageHeader`
- `SectionHeader`
- `FilterBar`

### 13.2 组件设计要求

每个组件都应同时定义：

- 默认态
- hover 态
- active 态
- disabled 态
- loading 态
- error 态

不能只设计静态图。

### 13.3 组件一致性规则

- 同类按钮全站高度一致
- 同类输入框全站边框、圆角、聚焦态一致
- 同类卡片全站阴影和边框语言一致
- 同类标题区全站间距一致

### 13.4 组件抽象分层

建议按三层治理组件，避免重复造轮子：

- 基础层：
  - `Button`
  - `Input`
  - `Tag`
  - `Card`
  - `Modal`
- 模板层：
  - `PageHeader`
  - `SectionHeader`
  - `FilterBar`
  - `ListPageLayout`
  - `DetailPageLayout`
- 业务层：
  - `PostCard`
  - `FoodCard`
  - `MerchantCard`
  - `PublishEntryCard`

规则是：

- 业务层尽量依赖模板层和基础层
- 模板层尽量依赖基础层
- 不要让业务层互相复制结构

## 14. 模块级风格差异边界

可以有差异，但不能散：

- 广场更开放、轻快
- 食堂更温暖、食欲感
- 社团更活跃、色彩感更强
- 指南更安静、可信
- 后台更克制、信息效率优先

但这些差异只能体现在：

- 模块色
- Banner 氛围
- 标签色
- 局部插图

不能体现在：

- 完全不同的按钮体系
- 完全不同的圆角和阴影
- 完全不同的布局骨架

## 15. 文案语气与界面语调

界面文案风格建议：

- 年轻
- 清楚
- 友好
- 不卖弄
- 不企业黑话

例如：

- 用“发布帖子”而不是“提交内容”
- 用“保存修改”而不是“确认提交”
- 用“还没有内容，去看看别人的分享”而不是“暂无数据”

语气像校园产品，而不是行政系统。

## 16. 技术落地建议

### 16.1 先做 Token 层

建议优先建立：

- 颜色 token
- 字体 token
- 间距 token
- 圆角 token
- 阴影 token
- 动效 token

### 16.2 再做基础组件层

在 Web 侧先统一：

- 通用卡片
- 通用按钮
- 通用表单控件
- 通用空态和错误态
- 通用页面头部

### 16.3 再做页面模板层

在视觉语言统一后，再套进：

- 列表页模板
- 详情页模板
- 表单页模板
- 工作台模板
- 后台模板

### 16.4 最后做模块视觉迁移

按模块逐步迁移：

- 广场
- 食堂
- 我的
- 社团 / 二手 / 跑腿 / 指南
- 后台

## 17. 验收标准

这份视觉方案生效后，后续页面评审至少要满足：

1. 用户能一眼认出这是同一个产品
2. 不同页面之间不会像来自不同模板
3. 模块有区分，但不会风格割裂
4. 动效统一、克制、舒服
5. 浅色调、校园风、活泼感稳定成立
6. 组件状态完整，不只看静态截图
7. 没有明显重复组件和平行轮子

## 18. 待确认问题

以下问题会影响后续视觉和组件治理方案，建议你后续逐项确认：

1. Web 端是否接受引入新的 Web 字体方案，还是必须完全使用现有字体体系
2. 首页是否需要保留当前启动视频 / 安装引导这类偏 PWA 的体验表达
3. Web 端是否希望保留部分“移动端感”的品牌记忆，例如圆角、悬浮入口、轻卡片漂浮感
4. 后台页面是否也要完全纳入同一套校园风体系，还是后台允许更克制、更偏管理台
5. 后续组件沉淀是否希望落到单独目录，例如 `frontend/src/components/ui` 或 `frontend/src/features`

## 19. 结论

这套 Web 视觉语言的核心不是“做得好看一点”，而是建立一套长期可维护的校园产品审美秩序。

统一方向可以概括为：

- 浅色晨光背景
- 纸张感卡片系统
- 彩色模块标签
- 轻快但克制的动效
- 友好清爽的校园公共空间气质

后续只要新页面、新组件都遵守这套语言，Web 改版就不会再出现一页一个风格的问题。
