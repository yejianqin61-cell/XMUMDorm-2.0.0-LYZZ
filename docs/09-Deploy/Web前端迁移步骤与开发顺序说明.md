
# Web 前端迁移步骤与开发顺序说明

## 1. 文档目标

本文档专门回答两个问题：

- Web 前端正式迁移时，应该按什么步骤推进
- 开发顺序应该怎么排，才能既稳又不返工

本文档默认前提：

- `frontend/` 是 Web 工作区
- `frontend-app/` 是 App / Capacitor 工作区
- `shared/*` 是双端共享层
- 本轮主要治理 Web，不主动破坏 App

## 2. 迁移总原则

### 2.1 先搭骨架，再迁页面

顺序必须是：

1. 先定视觉语言
2. 先定 token
3. 先定基础组件
4. 先定页面模板
5. 再逐模块迁移页面

不能反过来先一页一页硬改。

### 2.2 先复用，再新增

每次开发前统一先问：

- 现有组件能不能复用
- 现有模板能不能扩展
- 现有样式 token 能不能覆盖

只有不能复用时，才允许新增组件或模板。

### 2.3 先高频路径，再长尾页面

优先迁移：

- 首页 / 广场
- 帖子流
- 食堂核心链路
- 我的页面主入口

后迁移：

- 长尾发布页
- 次级工具页
- 后台长尾页面

### 2.4 一次只做一层变化

每个提交尽量只做一类事情：

- 只做 token
- 只做基础组件
- 只做某个模板
- 只做某个模块迁移

不要一个 commit 同时混入：

- 视觉重构
- 路由重写
- 组件重命名
- 目录搬迁

## 3. 推荐迁移顺序

### Step 0 文档与边界确认

目标：

- 把现状、规范、边界、视觉方向先写清楚

已完成或应完成文档：

- `Web前端现状盘点表.md`
- `Web桌面化布局规范.md`
- `Web前端视觉语言统一设计方案.md`
- `Web改版分阶段任务清单.md`
- `Web改版开发边界说明.md`

验收：

- 团队对改版目标一致
- 不再纠结“先改哪页”

### Step 1 建立设计 Token 层

目标：

- 先把全站统一变量建立起来

建议内容：

- 色彩 token
- 字体 token
- 字号层级 token
- 间距 token
- 圆角 token
- 阴影 token
- 动效 token

输出结果：

- `frontend/src/styles/` 下统一 token 文件

为什么先做这一步：

- 不先统一 token，后面每迁一页都在重复定颜色和间距

### Step 2 建立基础 UI 组件层

目标：

- 建立真正可复用的基础组件

优先级建议：

1. `Button`
2. `Input`
3. `Textarea`
4. `Select`
5. `Tag`
6. `Badge`
7. `Card`
8. `Tabs`
9. `Modal`
10. `Toast`
11. `EmptyState`
12. `ErrorState`

要求：

- 每个组件都支持 variant
- 每个组件都补齐状态
- 新组件必须先检查是否已有近似组件

验收：

- 新页面开发不再直接从页面里手搓按钮和卡片

### Step 3 建立页面模板层

目标：

- 把页面结构模板收敛出来

优先模板：

1. `PageHeader`
2. `SectionHeader`
3. `FilterBar`
4. `ListPageLayout`
5. `DetailPageLayout`
6. `FormPageLayout`
7. `DashboardPageLayout`
8. `AdminPageLayout`

为什么这一步很关键：

- 它决定后续页面迁移是“套模板”，而不是“重画一遍”

### Step 4 替换 Web 主壳

目标：

- 用桌面 Web 壳替换当前移动端 `Layout`

优先动作：

- 新建桌面 Header
- 新建左侧导航
- 新建右侧辅助栏
- 调整主内容容器
- 降低或移除底部 `TabBar` 的 Web 主导航职责

验收：

- Web 首页已经是网页，不再像放大的手机页面

### Step 5 迁移高频页面模板

先迁移最容易体现成果、同时复用价值最高的页面：

#### 5.1 列表页第一批

- `SquareTrendingList`
- `SquareCampusFeed`
- `MerchantList`
- `MyPosts`
- `MyReviews`

#### 5.2 详情页第一批

- `PostDetail`
- `FoodDetail`
- `MarketplaceDetail`
- `ClubProfile`

#### 5.3 表单页第一批

- `PostNew`
- `MarketplacePublish`
- `PublishErrand`
- `ProfileEdit`
- `FoodCreate`

目标：

- 用少量页面验证模板是否够稳

### Step 6 迁移核心模块

推荐顺序：

1. 广场 / 树洞 / 帖子链路
2. 食堂链路
3. 我的页面链路
4. 社团 / 二手 / 跑腿 / 新生指南
5. 后台

原因：

- 先做主流量模块，收益最大
- 先做结构相对清晰模块，模板最容易稳定
- 后台放后面，是因为它本身可以相对独立治理

### Step 7 清理旧结构与长尾样式

目标：

- 迁完主要页面后，回头清理重复结构和遗留样式

包括：

- 老旧页面级 CSS
- 重复按钮 / 卡片 / header
- `pages/` 目录里的伪页面组件
- 无用样式与废弃组件

验收：

- 代码树比现在更清晰
- 重复轮子明显减少

## 4. 开发顺序建议

建议采用下面的固定顺序执行每一轮开发：

1. 明确本轮只解决一个层级问题
2. 先查现有组件和模板是否可复用
3. 先做最小可复用抽象
4. 再改一个代表性页面验证
5. 验证通过后，再批量迁移同类页面
6. 每完成一个 task 提交一次

一个健康的开发节奏应该像这样：

```text
Token -> Base UI -> Page Template -> 代表页验证 -> 模块批量迁移 -> 清理收尾
```

而不是：

```text
首页改一点 -> 食堂改一点 -> 又回去改按钮 -> 又回去改颜色 -> 最后全站不统一
```

## 5. 单个页面的迁移步骤

每迁移一个页面，建议固定执行以下流程：

1. 判断它属于哪一类模板
2. 判断是否已有可复用的基础组件和业务组件
3. 先替换布局骨架
4. 再替换卡片、按钮、标签、筛选器等局部组件
5. 最后补动效、空态、错误态、响应式

不要上来先抠颜色和细节。

## 6. 单个组件的迁移步骤

每新增或重构一个组件，固定先问：

1. 是否已有相近组件
2. 是否只是样式差异
3. 能否抽成 variant
4. 是否值得沉淀为基础组件
5. 是否真的会被多处复用

如果答案不明确，就先不要新建。

## 7. 提交顺序建议

建议每次提交尽量小而清晰：

1. `docs(web): ...`
2. `feat(web-ui): add shared button and card primitives`
3. `feat(web-layout): add desktop list page template`
4. `refactor(web): migrate square trending to shared list layout`
5. `refactor(web): migrate merchant list to shared list layout`

不要把“基础建设”和“多个模块改版”揉成一个大提交。

## 8. 每阶段的验收口径

### 文档阶段

- 文档完整
- 方向一致

### Token 阶段

- 样式变量可用
- 不再页面硬编码主色和间距

### 基础组件阶段

- 新页面可直接复用基础组件
- 无明显重复轮子

### 模板阶段

- 同类页面能共用一套结构

### 模块迁移阶段

- 关键链路可用
- 视觉统一
- 页面不回归

### 收尾阶段

- `npm run build:web` 通过
- 遗留样式减少
- 目录职责更清晰

## 9. 待你确认的问题

下面这些点如果你给出明确答案，后续迁移顺序会更稳：

1. Web 首页第一批优先要改的是 `SquareHome`，还是希望先从登录页 / 我的页做一版样板
2. 当前是否接受先新建一套 Web 专属 `ui` 基础组件目录
3. 后续是否接受把 `pages/` 中的片段组件逐步迁移到 `components/` 或 `features/`
4. 组件命名你更倾向于“通用名 + variant”，还是“模块名 + 业务名”的保守方式
5. 你是否希望后台页面与主站同步统一视觉，还是后台允许单独保持更管理台的风格

## 10. 结论

前端迁移最怕的不是工作量大，而是顺序错。

正确顺序应该始终是：

- 先统一语言
- 先统一基础
- 先统一模板
- 再迁移页面
- 最后清理遗留

只要按这个顺序推进，后面每一轮开发都会越来越快，而不是越改越乱。
