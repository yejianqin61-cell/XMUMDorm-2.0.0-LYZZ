# Step5-高频页面迁移Task拆解文档

## 1. 文档目标

本文用于把 `Step5-高频页面迁移详细设计文档.md` 继续拆成可执行 task，方便后续严格按：

- 一次只做一个 task
- 一个 task 一个提交
- 代码类 task 完成后执行 `npm run build:web`

这套工作流推进。

## 2. 拆解原则

Step 5 的 task 拆解遵循以下原则：

1. 先做代表页，再做同类扩散。
2. 先做结构问题最明显的页面，再做次级页面。
3. 允许把强相关页面合并为一个 task，但不混入不相干模块。
4. 一个 task 只解决一个页面类型问题。

## 3. 首批 Task 列表

当前建议 Step 5 首批拆成以下 task：

1. `T28-TreeHole内容流桌面化验证.md`
2. `T29-Post搜索与标签流桌面化验证.md`
3. `T30-CanteenHome门户桌面化验证.md`
4. `T31-高频详情页桌面化二次验证.md`
5. `T32-Mailbox次级高频页桌面化验证.md`
6. `T33-个人列表页桌面化二次验证.md`

## 4. 各 Task 职责说明

### T28

解决：

- `TreeHole` 在主壳下仍偏移动端沉浸内容流的问题

验证重点：

- 主列节奏
- 工具栏落点
- 推荐块与内容流关系

### T29

解决：

- `PostSearch` 与 `PostTagFeed` 这类搜索/标签列表页在桌面壳下的统一问题

验证重点：

- 搜索区
- 标签筛选区
- 列表信息密度

### T30

解决：

- `CanteenHome` 这种模块门户首页的桌面首页节奏问题

验证重点：

- 首屏层次
- 模块入口区块
- 推荐区与工具区关系

### T31

解决：

- `FoodDetail` 与 `MerchantFoodDetail` 在桌面壳下的详情主轴一致性问题

验证重点：

- 正文宽度
- 元信息区位置
- 操作区与补充区分层

### T32

解决：

- `Mailbox` 在“我的”链路下的桌面次级高频页节奏问题

验证重点：

- 与 `MyZone` 的连续性
- 列表与详情或消息流结构
- 次级工具区落点

### T33

解决：

- `MyPosts` 与 `MyReviews` 在主壳下的个人列表页一致性问题

验证重点：

- 页面头部
- 列表区
- 空态/筛选/个人上下文节奏

## 5. 执行顺序建议

建议严格按下面顺序推进，不跳步：

```text
T28
-> T29
-> T30
-> T31
-> T32
-> T33
```

原因是：

- `T28` 先拿下最能体现桌面化变化的内容流页
- `T29` 把内容流和搜索列表页进一步串起来
- `T30` 再验证门户首页型页面
- `T31` 再做详情页二次验证
- `T32` 和 `T33` 最后收“我的”次级高频链路

## 6. 每个 Task 的统一验收目标

每个 Step 5 task 完成后，至少应达到：

1. 页面在主壳下的桌面结构比之前更稳定。
2. 页面移动端全屏假设明显下降。
3. 页面没有引入新的壳层职责冲突。
4. 页面主要业务功能不回归。
5. `npm run build:web` 通过。

## 7. 提交建议

Step 5 建议优先使用以下提交前缀：

- `docs(web-migrate): ...`
- `refactor(web-page): ...`
- `refactor(web-feed): ...`
- `refactor(web-detail): ...`

## 8. 结论

Step 5 的关键不在于“一口气改很多页”，而在于把高频页面桌面化迁移拆成一串低风险、可连续执行的小任务。

只要按这组 task 顺序推进，后续 Step 6 的模块级批量治理就会明显更顺。
