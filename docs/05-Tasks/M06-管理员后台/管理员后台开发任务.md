  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,

  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_status (status),

-- 用户封禁/禁言记录表

```js
// requireAdmin 中间件：在 authenticateToken 之后调用，校验 admin 角色
function requireAdmin(req, res, next) {
  if (req.user?.role !== 'admin') {
    return res.status(403).json({ status: -1, message: '需要管理员权限' });
  }
  next();
}
```

## 3. 模块 C：用户管理

### 3.1 需求描述

管理员查看用户列表、用户详情，执行封禁/禁言/解除操作。

### 3.2 后端 API

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/admin/users?page=&pageSize=&search=&role=&status=` | 用户列表（分页、搜索、筛选） |
| `GET` | `/api/admin/users/:id` | 用户详情（信息+发帖数+评论数+举报次数+封禁/禁言记录） |
| `POST` | `/api/admin/users/:id/ban` | 封禁用户（body: `{ duration: 1|3|7|30|null }`，null=永久） |
| `POST` | `/api/admin/users/:id/unban` | 解除封禁 |
| `POST` | `/api/admin/users/:id/mute` | 禁言用户（body: `{ duration }`） |
| `POST` | `/api/admin/users/:id/unmute` | 解除禁言 |
| `DELETE` | `/api/admin/users/:id` | 删除账号（软删除或标记 deactivated） |

### 3.3 封禁/禁言生效机制

- 每次请求时，在 `authenticateToken` 之后插入 `checkSanction` 中间件
- 若用户有活跃封禁 → 返回 403「账号已被封禁至 YYYY-MM-DD」
- 若用户有活跃禁言 → 发帖/评论接口返回 403「账号已被禁言至 YYYY-MM-DD」
- 封禁/禁言过期后自动失效（按 `ends_at < NOW()` 判定，无需定时任务）

### 3.4 users 表扩展

需要在 `users` 表增加字段（或在 `user_sanctions` 表查当前状态）：

建议在 `users` 表增加：
- `status` ENUM('active','banned','deactivated') DEFAULT 'active'
- `banned_until` DATETIME NULL
- `muted_until` DATETIME NULL

新建迁移 `migrations/057_user_status.sql`。

### 3.5 前端页面

**新建文件：**

### 4.4 自动举报规则

在内容被举报时触发（`POST /api/reports` 后置逻辑）：
- 同一内容被举报 ≥ 3 次 → 自动隐藏（参考现有 `hidden_by_admin` 等字段）
- 同一内容被举报 ≥ 5 次 → 自动下架（二手/跑腿）
- 同一内容被举报 ≥ 10 次 → 自动进入审核队列（status 改为 processing）

## 13. 关键设计决策

### 13.1 前端路由命名约定

后台路由统一挂在 `/myzone/admin/` 下：
- `/myzone/admin` → Dashboard
- `/myzone/admin/users` → 用户列表
- `/myzone/admin/users/:id` → 用户详情
- `/myzone/admin/reports` → 举报列表
- `/myzone/admin/reports/:id` → 举报详情
- `/myzone/admin/contents/:module` → 内容列表
- `/myzone/admin/contents/:module/:id` → 内容详情
- `/myzone/admin/announcements` → 公告管理
- `/myzone/admin/config` → 系统配置
- `/myzone/admin/sensitive-words` → 敏感词管理
- `/myzone/admin/logs` → 操作日志

### 13.2 权限校验策略

**双重校验：**
- 前端：`AuthGuard` + `isAdmin` 条件渲染（防止非管理员看到后台入口）
- 后端：`authenticateToken` + `requireAdmin` 双中间件（防止非管理员调用 API）

即使前端路由被绕过，后端仍会拒绝非 admin 角色的请求。

### 13.3 移动端适配原则

- 侧边栏在移动端收缩为汉堡菜单（屏幕宽度 < 768px 时默认隐藏）
- 表格在移动端改为卡片布局（每行数据渲染为一张卡片）
- 操作按钮在移动端使用底部固定操作栏

### 13.4 内容管理通用化策略

8 个内容模块使用同一套 `ContentList` + `ContentDetail` 组件，通过 URL 参数 `:module` 区分：
- 组件内维护 `moduleConfig` 映射（字段名、搜索字段、操作项等）
- 避免为每个模块重复编写相似的列表/详情页
