# 等级系统开发计划（V1）

**依据设计文档：**

- `docs/02-设计文档/Dorm 等级系统设计文档（V1）.md`（文档内标题为 V3.0，以下统称「设计文档」）

**项目：** Dorm · XMUMDorm-2.0.0-LYZZ  
**编写目的：** 将等级 / 经验 / 徽章体系拆解为可执行的前后端任务，明确与现有模块（登录、树洞、广场热搜、食堂点评）的埋点位置与防刷策略。  
**核心定位：** 用行为激励提升活跃与留存；**V1 不做**连续签到、周榜、成就系统（设计文档 §9 后续扩展）。

| 设计模块 | 现状 | 缺口等级 |

### 0.3 模块边界（避免 scope 膨胀）

| 内容 | V1 职责 | 不在 V1 |

| 树洞 + 广场热搜 点赞/评论/发帖 | ✅ 埋点 | 校园此刻帖是否计经验（建议 V1.1 与设计方确认，默认**计入发帖/互动**） |
| 食堂普通/优质点评 | ✅ 埋点 | 店铺评分「夯爆了」等级（那是菜品评分，不是用户等级） |

### 0.4 等级与经验常量（与设计文档对齐）

设计文档「累计经验」列为**达到该等级所需的最低累计经验**（注册默认为 Lv1、`exp=0`）：

| 等级 | 名称 | 累计经验（≥ 即为该级） |
|------|------|------------------------|
| Lv1 | Freshman（新生） | 0 |
| Lv2 | Explorer（探索者） | 100 |
| Lv3 | Contributor（贡献者） | 300 |
| Lv4 | Campus Star（校园达人） | 800 |
| Lv5 | Senior（资深成员） | 1800 |
| Lv6 | Legend（校园传奇） | 4000 |
| （满级） | — | 8000（达 Lv6 上限，可继续累计展示） |

> **实现注意：** 设计文档最高档 Lv6 对应累计 8000；`LEVEL_THRESHOLDS = [0, 100, 300, 800, 1800, 4000, 8000]`，升级判定取 `exp` 所在区间。若产品确认 Lv6 为终点且不再升级，8000 以上仍显示 Lv6。

## 1. 目标能力（V1）

### 1.1 用户侧体验

```
用户行为（登录/点赞/评论/发帖/点评）
        ↓
  后端 grantExp（校验字数/图片/每日上限/防刷）
        ↓
  users.exp 增加；写 exp_logs
        ↓
  若跨越等级阈值 → level 更新 + badge 更新
        ↓
  响应中带 { expDelta, levelUp?, newLevel?, badge? }
        ↓
  前端：Toast（+N 经验）/ 升级全屏弹窗 / 优质帖奖励提示
```

### 1.2 经验规则汇总（实现配置）

| action | 单次经验 | 每日上限 | 附加条件 |
|--------|---------:|---------:|----------|
| `login` | +5 | 5 | 每日首次有效登录 |
| `like` | +1 | 15 | 树洞帖或热搜帖点赞；取消赞回退 |
| `comment` | +5 | 15 | 树洞/热搜评论，字数 ≥ 5 |
| `post` | +10 | 15 | 树洞/热搜发帖，内容 ≥ 10 字 |
| `cafeteria_review` | +10 | 30 | 食堂普通点评 |
| `quality_bonus` | +5 | 15 | 点评字数≥20 **或** 有图；与普通叠加 |
| `post_popular_like` | +20 | — | 单帖点赞数首次≥10，每帖一次 |
| `post_popular_comment` | +20 | — | 单帖评论数首次≥10，每帖一次 |
| `level_up` | 0 | — | 流水记录，不增加 exp |

### 1.3 展示位置（前端）

| 位置 | 展示内容 |
|------|----------|
| 我的（`MyZone`） | 头像下：`👑 Lv6` + 经验条（当前级进度） |
| 用户主页（若有 `/user/:id`） | 同上 |
| 帖子卡片 `PostCard` | 昵称旁等级徽章 |
| 详情/评论 `PostDetailShell`、评论列表 | 评论者昵称旁 `⭐ Lv4` |
| 升级 | 全局 `LevelUpModal`（动画 + 新徽章名） |

## 2. 数据模型（V1）

### 2.1 `users` 表扩展

| 字段 | 类型 | 说明 |
|------|------|------|
| `level` | TINYINT NOT NULL DEFAULT 1 | 当前等级 1~6 |
| `exp` | INT NOT NULL DEFAULT 0 | 累计经验 |
| `badge` | VARCHAR(32) NULL | 当前徽章 key，如 `freshman`；可由 level 推导，冗余存储便于查询 |
| `exp_today_total` | INT NOT NULL DEFAULT 0 | 当日已获得经验（可选，或用流水聚合） |
| `exp_today_date` | DATE NULL | 上次重置每日计数的日期（上海时区自然日） |

> 注册时：`level=1, exp=0, badge='freshman'`。

### 2.2 `exp_logs` 经验流水

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | |
| `user_id` | INT NOT NULL | FK users |
| `action` | VARCHAR(32) NOT NULL | 见 §1.2 |
| `exp` | INT NOT NULL | 正为增加，负为回退 |
| `ref_type` | VARCHAR(32) NULL | 如 `post` / `product_comment` / `trending_post` |
| `ref_id` | BIGINT NULL | 关联实体 id |
| `meta_json` | JSON NULL | 如 `{ dailyBucket: 'like', reason: 'unlike' }` |
| `created_at` | TIMESTAMP | |

索引建议：`(user_id, created_at)`、`(user_id, action, created_at)`（查每日上限）。

### 2.3 `user_exp_daily` 每日分桶（推荐）

比仅用 `exp_today_total` 更易维护各行为独立上限：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | INT | |
| `day` | DATE | 上海时区自然日 |
| `action` | VARCHAR(32) | `login` / `like` / `comment` / … |
| `exp_sum` | INT | 该行为当日已累计经验 |

PRIMARY KEY (`user_id`, `day`, `action`)。

### 2.4 `post_exp_rewards` 帖子一次性奖励

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | |
| `post_kind` | ENUM | `treehole` / `trending`（树洞 posts vs trending_posts） |
| `post_id` | BIGINT | |
| `reward_type` | ENUM | `likes_10` / `comments_10` |
| `user_id` | INT | 帖主 |
| `granted_at` | TIMESTAMP | |

UNIQUE (`post_kind`, `post_id`, `reward_type`)。

## 3. 后端设计

### 3.1 核心服务 `services/expService.js`

| 函数 | 职责 |
|------|------|
| `getLevelByExp(exp)` | 根据累计经验返回 level |
| `getBadgeByLevel(level)` | 返回徽章 key + 展示名 |
| `grantExp(userId, { action, amount, refType, refId, meta })` | 事务内：校验上限 → 写 log → 更新 users.exp → 升级检测 → 返回结果 |
| `revokeExp(userId, logId or ref)` | 取消赞等场景回退 |
| `checkAndGrantPostPopularRewards(postKind, postId, authorId)` | 统计赞/评数，达标且未发过则 +20 |
| `resetDailyBucketsIfNeeded(userId)` | 跨日清零 `user_exp_daily` |

**原则：** 所有经验变动只走此服务，禁止路由内直接 `UPDATE users SET exp`。

### 3.2 埋点位置

| 路由文件 | 触发点 | action |
|----------|--------|--------|
| `routes/auth.js` | `POST /login` 成功 | `login` |
| `routes/posts.js` | 发帖成功 | `post`（校验字数） |
| `routes/posts.js` | 评论成功 | `comment` |
| `routes/posts.js` | 点赞 toggle 成功 | `like` / 取消则 `revoke` |
| `routes/square.js` | 热搜发帖 / 评论 / 点赞 | 同上 |
| `routes/canteen.js` | `POST` 商品点评成功 | `cafeteria_review` + 可选 `quality_bonus` |
| `routes/posts.js` / `square.js` | 点赞/评论后异步或同事务 | `checkAndGrantPostPopularRewards` |

### 3.3 API 扩展

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/users/me/level` | 当前 level、exp、badge、下一级所需经验、今日各桶用量（可选） |
| GET | `/api/users/:id/level` | 公开资料页展示 |
| GET | `/api/users/me/exp-logs` | 分页流水（V1 可选，MyZone 二级页） |

**现有接口扩展：** `GET /api/users/profile`（或登录响应）增加 `level, exp, badge`，减少额外请求。

### 3.4 防刷规则（实现清单）

| 规则 | 实现方式 |
|------|----------|
| 每日上限 | `user_exp_daily` 插入前检查 `exp_sum + delta <= cap` |
| 评论/发帖字数 | 服务端 `trim` 后计字符（中英文均计），不满足则 `expDelta: 0` 但业务仍成功 |
| 重复点赞无效 | toggle 为 unlike 时 `revokeExp`；同一用户对同一帖重复 like 不重复加 |
| 自己给自己点赞/评论 | **建议不加经验**（V1 明确写入代码） |
| 优质帖奖励每帖一次 | `post_exp_rewards` UNIQUE 约束 |
| 删除帖/删评 | V1 **不回溯**已发经验（简单）；V1.1 可考虑扣回 |

业务接口成功时可选附带：

```json
{
  "status": 0,
  "data": { "...": "原业务数据" },
  "exp": {
    "delta": 10,
    "total": 760,
    "level": 3,
    "levelUp": false,
    "badge": "contributor",
    "messages": []
  }
}
```

`levelUp: true` 时附加 `previousLevel`、`newLevel`、`badgeName`，供 `LevelUpModal` 使用。

| 类型 | 内容 |
|------|------|
| 升级 | exp 从 99→100 触发 Lv2；弹窗与 badge 更新 |

## 8. 风险与依赖

| 风险 | 缓解 |
|------|------|
| 埋点遗漏导致经验不一致 | 统一 `expService`；代码审查清单对照 §3.2 |
| 每日上限时区 | 统一用上海自然日，与待办/排行现有逻辑一致 |
| 升级弹窗打扰 | 仅 `levelUp` 时全屏；普通 `+N` 用轻 Toast |
| 老帖突然触发「热门奖励」 | 上线时只对**新达标**事件触发；或一次性脚本扫库需产品确认 |
| 性能（每次点赞查 COUNT） | 热门奖励可异步；或维护 `like_count` 冗余字段 |
| 设计文档防刷 §7 未写全 | V1 按 §3.4 实现；未决项记入 V1.1 |

**外部依赖：** 登录态（JWT）、现有帖子/点评表结构稳定。
