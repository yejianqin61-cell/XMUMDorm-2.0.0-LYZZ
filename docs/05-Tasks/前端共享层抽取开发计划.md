# 前端共享层抽取开发计划

## 背景

当前仓库前端已拆分为：

- `frontend/`：Web 端工作区
- `frontend-app/`：Capacitor App 工作区

两套前端继续共用：

- 同一套后端
- 同一套 API 接口
- 同一套业务数据结构

当前问题不在后端，而在于前端拆分后，`frontend` 与 `frontend-app` 中仍保留了大量完全相同的接口层、工具层与状态层代码。若不及时抽取，共享逻辑会在后续开发中发生双份维护、双份修复、双份回归验证。

## 盘点结论

基于当前仓库实际比对，以下目录在 `frontend/src` 与 `frontend-app/src` 中目前是 100% 一致的：

| 目录 | Web 文件数 | App 文件数 | 当前差异 |
|---|---:|---:|---|
| `api/` | 20 | 20 | 0 |
| `utils/` | 11 | 11 | 0 |
| `constants/` | 3 | 3 | 0 |
| `config/` | 1 | 1 | 0 |
| `context/` | 5 | 5 | 0 |
| `query/` | 2 | 2 | 0 |
| `services/` | 2 | 2 | 0 |
| `i18n/` | 2 | 2 | 0 |
| `data/` | 5 | 5 | 0 |

结论：

- 当前最适合抽取共享层的，不只是 API。
- API、请求配置、纯工具、查询配置、业务常量，已经满足“立即抽取”的条件。
- Context、服务层、i18n、mock data 也可以共享，但应分批进行，避免一次性修改面过大。

## 抽取目标

建立统一共享目录，例如：

```text
shared/
  api/
  config/
  utils/
  constants/
  query/
  context/
  services/
  i18n/
  data/
```

然后让：

- `frontend/` 通过别名或相对路径引用共享层
- `frontend-app/` 通过别名或相对路径引用共享层

共享目标是“逻辑层统一”，不是“页面层统一”。

## 第一批必须抽取

这一批收益最高、风险最低，应优先完成。

### 1. `shared/api/`

这是必须共享的核心层。原因：

- 直接对接同一套后端接口
- 两端接口契约必须保持一致
- 出现接口变更时，不应改两套文件

应抽取文件：

- `admin.js`
- `auth.js`
- `canteen.js`
- `clubs.js`
- `config.js`
- `diary.js`
- `errands.js`
- `handbook.js`
- `marketplace.js`
- `notifications.js`
- `organizations.js`
- `posts.js`
- `push.js`
- `rankings.js`
- `request.js`
- `schedule.js`
- `square.js`
- `tags.js`
- `todos.js`
- `users.js`

说明：

- `request.js` 与 `config.js` 虽然在 `api/` 下，但实际上属于共享基础设施，应和业务 API 一并迁移。
- `request.js` 负责 token、统一错误处理、JSON 协议处理，是所有 API 的核心依赖。

### 2. `shared/constants/`

应抽取文件：

- `canteen.js`
- `levelConfig.js`
- `rating.js`

原因：

- 这些文件本质是业务枚举、评分阈值、静态配置。
- 它们不应随着 Web/App 页面分化而各自维护。

### 3. `shared/utils/` 中的纯工具

第一批应优先抽取以下稳定纯工具：

- `apiError.js`
- `formatTime.js`
- `formatTodoDue.js`
- `nestComments.js`
- `regionCode.js`
- `schedulePersist.js`
- `scrollCache.js`

说明：

- 这些工具主要承担数据处理、错误归一、轻量状态持久化。
- 与页面布局无关，跨端共享收益高。

### 4. `shared/query/`

应抽取文件：

- `queryClient.js`
- `queryKeys.js`

原因：

- Web 与 App 使用相同的 TanStack Query 策略
- Query key 与缓存策略本来就应统一管理

## 第二批建议抽取

这一批当前也完全一致，但和运行环境、表现层、交互层耦合更高，建议在第一批完成并稳定后再抽。

### 1. `shared/utils/` 中的环境相关工具

建议第二批抽取：

- `capacitor.js`
- `fullscreen.js`
- `imagePicker.js`
- `motion.js`

原因：

- 它们虽是工具，但和 Capacitor、浏览器全屏、动画策略、文件选择器等前端运行环境强相关。
- 未来 Web 与 App 可能出现不同实现，因此宜晚于纯工具抽取。

### 2. `shared/context/`

建议第二批抽取：

- `AuthContext.jsx`
- `ExpFeedbackContext.jsx`
- `LanguageContext.jsx`
- `ToastContext.jsx`
- `Toast.css`

原因：

- 这些文件当前逻辑一致，且依赖同一套 API 与 storage 方案。
- 但它们已经开始接近应用壳层，尤其 `Toast.css` 未来大概率会随 Web 改版而变化。
- 建议先共享 JS 逻辑，再评估样式是否继续共用。

### 3. `shared/services/`

建议第二批抽取：

- `jpushService.js`
- `pushService.js`

原因：

- 推送能力是平台级能力，原则上应该统一。
- 但它与 Capacitor、浏览器 Service Worker、极光推送策略相关，适合在 API 层稳定后再纳入共享层。

### 4. `shared/i18n/`

建议第二批抽取：

- `canteenAreaRanking.js`
- `canteenStrings.js`

原因：

- 文案与业务一致时适合共享。
- 若未来 Web 改版需要不同表达风格，可再拆回各端覆盖。

### 5. `shared/data/`

建议第二批抽取：

- `mockCanteen.js`
- `mockComments.js`
- `mockNotifications.js`
- `mockPosts.js`
- `mockRankings.js`

原因：

- 这批文件是 mock 数据，本身天然适合共用。
- 但它们不影响生产主链路，因此优先级低于 API 与工具层。

## 暂不抽取

以下内容当前不建议进入共享层：

- `pages/`
- `routes/`
- `components/`
- `styles/`
- 大多数 `App.jsx`、`Layout.jsx`、导航、Header、TabBar、页面壳组件

原因：

- Web 与 App 接下来会在布局、导航、交互、视觉体系上分化
- 这些层共享会重新把两端耦合回去
- 未来变更频率高，强共享会显著拖慢开发

## 推荐实施顺序

### 阶段 1：建立共享目录与别名

目标：

- 新建 `shared/`
- 在 `frontend/vite.config.js` 与 `frontend-app/vite.config.js` 中增加共享别名，例如 `@shared`

产出：

- 两端都可从共享目录导入模块
- 不改业务逻辑，只搭建路径基础设施

### 阶段 2：抽取第一批共享层

目标：

- 抽 `api/`
- 抽 `constants/`
- 抽纯工具 `utils/`
- 抽 `query/`

产出：

- 两端完全共用同一套接口访问逻辑
- 接口改动只改一处

### 阶段 3：构建与回归验证

必须验证：

- `npm run build:web`
- `npm run build:app`
- `npm run build:capacitor`

建议验证页面：

- 登录注册
- 树洞帖子列表与详情
- 广场
- 食堂
- 课程表
- 通知
- 用户中心

### 阶段 4：抽取第二批共享层

目标：

- 逐步抽 `context/`
- 抽 `services/`
- 抽环境相关 `utils/`
- 抽 `i18n/`
- 抽 `data/`

原则：

- 每一类抽完都单独构建与回归
- 不建议一次性把第二批全搬完

## 依赖关系与注意事项

### 1. `api/request.js` 是共享层核心入口

它依赖：

- `api/config.js`
- 浏览器 `fetch`
- `localStorage`

说明：

- 由于当前 App 是 Capacitor 打包 Web，因此 Web 与 App 都具备这些能力
- 这也是 API 层现在可以安全共享的根本原因

### 2. `AuthContext.jsx` 依赖 API 与 storage

它依赖：

- `api/config.js`
- `api/users.js`
- `utils/apiError.js`
- `localStorage`
- `sessionStorage`

说明：

- 当前可以共享
- 但如果未来 App 想切换更原生的凭证存储方案，再考虑做端侧适配层

### 3. `pushService.js` 与 `jpushService.js` 需保留平台守卫

说明：

- 共享后仍必须保留 `isNative()`、Capacitor 检测、Web fallback
- 不允许为了共享而去掉平台分支

## 风险评估

### 低风险

- `api/`
- `constants/`
- `query/`
- 纯 `utils/`

### 中风险

- `context/`
- `i18n/`
- `data/`

### 中高风险

- `pushService.js`
- `jpushService.js`
- `capacitor.js`
- `imagePicker.js`
- `fullscreen.js`
- `motion.js`

## 预估工程量

按本项目当前状态，建议分两次完成：

### 第一次交付

范围：

- `api`
- `constants`
- 纯 `utils`
- `query`

预估：

- 0.5 到 1 天

### 第二次交付

范围：

- `context`
- `services`
- 环境相关 `utils`
- `i18n`
- `data`

预估：

- 0.5 到 1.5 天

合计预估：

- 1 到 2.5 天

## 最终建议

建议按以下优先级执行：

1. 立即共享 `api + request + config + constants + 纯 utils + query`
2. 暂缓页面、组件、路由、样式层共享
3. 等 Web 大改开始后，再视差异程度决定是否共享 `context / services / i18n`

这套方案既能保证后端接口层统一，又不会阻碍 Web 与 App 在表现层的正式分化。
