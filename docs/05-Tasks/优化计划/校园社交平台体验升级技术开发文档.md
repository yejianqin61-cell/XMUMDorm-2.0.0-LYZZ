## 三、总技术策略

## 3.1 总体原则

本次升级继续遵循当前项目技术现实：

- 前端为 `Web + Capacitor` 双形态
- 不新增 RN 独立端实现
- 保持 React 单前端工程演进（react最近阶段不管他）
- 后端继续基于 Express + MySQL 扩展

### 3.2 改造方式

本次升级不建议“大重写”，建议采取以下路径：

1. 优先保留现有模块路由结构
2. 在现有页面基础上做首页重组和组件抽象
3. 将新增能力尽量设计为独立接口与独立表，避免破坏已有逻辑
4. 优先抽公共 UI/状态/卡片层，再逐页替换

### 3.3 技术分层

建议按四层推进：

1. 表现层
   - 页面布局
   - 卡片系统
   - 动效系统
   - Skeleton / 空态 / 错误态

2. 交互层
   - 首页聚合
   - 发布中心
   - 报名 / 通知 / 提醒
   - 关系推荐

3. 数据层
   - 聚合接口
   - 推荐接口
   - 活动报名接口
   - 通知分类接口

4. 存储层
   - 新增报名表
   - 新增个性化 / 行为记录表
   - 增强通知字段

## 五、P0 技术开发方案

## 5.1 P0-1 默认首页改为广场页

### 目标

- 进入应用后默认落在广场页
- 广场页承载“今日校园”首页能力
- Tab 栏第一个入口为广场

### 前端改动

涉及文件：

- `frontend/src/routes/layoutRoutes.jsx`
- `frontend/src/components/TabBar*`
- `frontend/src/pages/SquareHome.jsx`
- 可能涉及 `App.jsx` / Layout 默认 index 路由定义

### 实施建议

1. 将 layout index route 从 `TreeHole` 调整为 `SquareHome`
2. 保留树洞独立入口，但不再作为默认首页
3. 调整底部 Tab 的顺序与默认高亮逻辑
4. 检查首页返回逻辑与浏览器历史记录是否受影响

### 验收标准

- 登录后默认进入广场页
- 未登录访客进入后同样默认进入广场页
- Tab 高亮与页面一致

### 验收标准

- 广场首页首屏能展示“今日校园”摘要
- 不再只是模块入口拼接
- 首屏关键数据能一次接口返回，避免 5~8 个独立接口并发阻塞首屏

## 5.4 P0-4 动效系统基础版

### 目标

增加必要的页面切换、卡片出现、操作反馈动效。

### 前端改动

建议新增：

- `frontend/src/utils/motion.js`
- `frontend/src/components/ui/FadeInSection.jsx`
- `frontend/src/components/ui/RouteTransition.jsx`

### 推荐方案

当前项目已经使用 `framer-motion`，P0 继续沿用，不增加新动画库。

### P0 动效范围

1. 广场首页区块进入动画
2. 卡片分批进入
3. 点赞 / 收藏按钮微反馈
4. 发布成功 Toast + 列表高亮
5. Skeleton → 内容切换过渡

### 注意事项

- Capacitor 端需注意性能降级
- 保持动画时长在 `120ms ~ 260ms`
- 不在重列表上叠加高成本动画

### 验收标准

- 高频操作均有明确但克制的反馈
- 无明显掉帧和布局抖动

## 5.5 P0-5 发布中心基础版

### 目标

提供统一发布入口，提高内容供给效率。

### 前端改动

建议新增页面：

- `frontend/src/pages/PublishCenter.jsx`

建议新增组件：

- `frontend/src/components/publish/PublishEntryCard.jsx`
- `frontend/src/components/publish/PublishQuickActionSheet.jsx`

### 路由建议

- `GET /publish` 对应页面路由
- 从广场首页、Tab 中央按钮、部分模块头部跳转

### 首批入口内容

- 发树洞
- 发二手
- 发跑腿
- 发社团动态（注意鉴权）
- 发活动（注意鉴权）

### 后端改动

P0 阶段发布中心本身不一定新增后端接口，可先复用现有发布接口。

核心是前端统一入口，不改变原业务接口：

- 树洞：现有 posts 接口
- 二手：现有 marketplace 接口
- 跑腿：现有 errands 接口
- 社团动态 / 活动：现有 clubs 接口扩展

### 验收标准

- 用户能在一个入口内看到所有可发布内容
- 每个发布类型能正确跳转到对应发布页

## 5.6 P0-6 社团活动报名闭环基础版

### 目标

让活动具备最基本的：

- 查看详情
- 报名
- 取消报名
- 截止提醒
- 通知回流

### 数据库建议

新增表：

`club_activity_registrations`

建议字段：

```sql
id
activity_id
user_id
status              -- registered / cancelled
created_at
updated_at
cancelled_at
```

可增加唯一约束：

- `(activity_id, user_id)`

### 后端接口建议

1. `POST /api/clubs/activities/:id/register`
2. `DELETE /api/clubs/activities/:id/register`
3. `GET /api/clubs/activities/:id/registration-status`

可在活动详情接口中直接附带：

```json
{
  "registered": true,
  "registration_count": 23,
  "registration_deadline": "2026-06-30T18:00:00.000Z"
}
```

### 前端改动

涉及：

- `frontend/src/pages/Clubs/ActivityDetail.jsx`
- `frontend/src/components/clubs/ActivityRegisterBar.jsx`

### 通知联动

报名成功后可写入通知：

- 活动开始前 24 小时提醒
- 活动开始前 1 小时提醒

P0 先支持数据库通知写入，不要求复杂调度后台界面。

### 验收标准

- 用户能报名和取消报名
- 报名人数实时反映
- 报名状态可在详情页稳定显示

## 12.2 主要风险

### 风险 1：首页改动过大影响现有入口习惯

应对：

- 保留原模块入口
- 广场页内部重组，不删除主要功能

### 风险 2：UI 抽象过度导致改造成本过高

应对：

- 先只覆盖 P0 页面
- 不强行一次性替换全站

### 风险 3：活动闭环牵涉通知和状态，联调复杂

应对：

- 先做报名基础版
- 提醒采用最简单的通知写入方式

## 十三、结论

本技术开发文档建议按以下优先级执行：

### P0

- 默认首页切换到广场
- 广场页升级为“今日校园”
- 卡片系统 / 状态系统统一
- 动效系统基础版
- 发布中心基础版
- 活动报名闭环基础版
- 通知事务分类基础版

### P1

- 广场顶部个性化推荐
- 树洞 / 广场长列表节奏优化
- 个人主页校园身份增强
- 通知分栏与聚合
- 轻量关系链推荐

### P2

- 新生专区完整路径
- 活动签到 / 回顾
- 日历深度联动
- 完整个性化推荐
- 关系图谱扩展

这份顺序的核心原则是：

> 先改“第一眼体验”和“主闭环”，再做“个性化”和“校园纵深能力”。

如果按本文推进，团队可以先用 P0 在较短周期内做出一轮可感知的产品升级，再逐步向 P1、P2 扩展。
