🔴 关键技术偏差与风险

非 Expo Router 路由

HIGH

使用自定义 TabNavigator + 简易 router.ts 替代 Expo Router。导致：无法享受 Expo Router 的深度链接、类型安全路由、SSR 支持。与 Constitution 规定的 "Expo Router 文件系统路由" 偏离。

未使用 expo-image

HIGH

Constitution 和 rules/mobile.md 明确要求使用 expo-image 而非 React Native Image。当前所有屏幕使用标准 Image 组件，缺少渐进加载、缓存优化等能力。

ScrollView 替代 FlatList

MEDIUM

多数列表页面使用 ScrollView 而非 FlatList/FlashList。大列表场景 (树洞瀑布流、食堂菜品列表) 存在性能风险。虽然当前数据量可能不大，但不符合性能最佳实践。

大型 Screen 组件

MEDIUM

TreeholeScreen (17KB/400+ 行), ScheduleScreen (14KB), TodoScreen (13KB) 等文件过大，缺少组件拆分。hooks/ 目录为空，未抽取可复用逻辑。

六、隐藏风险与虚假进度 · Hidden Risks & False Progress

识别表面上"已完成"但实际存在深层问题的领域

1. 路由架构偏离宪法

HIGH

移动端约束.md 明确要求使用 Expo Router。实际代码使用自定义 TabNavigator + 简易 router.ts。当前"能用"但未来深度链接、类型安全路由、web/native 统一路由等均为不可用状态。这是架构债，越晚修越贵。

2. 大型组件未拆分

MEDIUM

TreeholeScreen (400+ 行) 包含瀑布流布局逻辑、Tag 筛选、列表渲染、发布按钮、搜索等功能于单一文件。hooks/ 目录为空意味着 0 个 Custom Hook 被抽取。代码复用度低，维护成本高。

3. UI 组件库不完整

MEDIUM

Web 端有 ~60 个共享组件。Mobile 端只有 5 个 UI 组件 + 4 个业务组件。大量 UI 元素在各 Screen 中内联实现。缺乏：SkeletonCard、SkeletonFood、LikeBurst、ImagePreview、UserLevelBadge、LevelUpModal 等。

4. API 层代码差异

LOW

mobile/src/api/ 有 21 个文件，而 Web 有 20 个。多出的 client.ts 和 request.js 是新增封装。虽然 Constitution 说 "API 层 100% 复用"，但实际是搬运+适配，API BASE URL 等问题需要差异处理。

5. 416 测试用例的真实覆盖

MEDIUM

20 个测试套件、416 用例、100% 通过 — 数字很漂亮。但需要审查：是否为浅层快照测试？是否覆盖了异步数据流？是否测试了边界条件和错误状态？100% 通过率在 416 个用例中有时意味着测试写得太简单。
