零、Capacitor 是什么？—— 科普章

5 分钟理解 Capacitor 的本质

一句话定义

Capacitor = 把你的 Web 前端装进一个原生 App 壳子，同时提供 JS 调用原生 API（相机、推送、文件等）的桥梁。

它由 Ionic 团队开发，是 Cordova 的精神继承者，但设计更现代、更符合 2026 年的 Web 标准。

和 PWA / React Native 的本质区别

维度PWACapacitorReact Native
渲染引擎 浏览器 系统 WebView（iOS=Safari, Android=Chrome） JS 驱动的原生 UI
UI 代码 HTML+CSS+JS HTML+CSS+JS（和 Web 同一份） JSX + RN 组件 API（不同的一套）
CSS 支持 完整 完整（backdrop-filter, grid, animation…） 有限的 StyleSheet API
App Store 分发 ❌ 不能 ✅ 可以 ✅ 可以
原生 API 有限的 Web API 插件系统（相机/推送/文件/Haptics…） 原生模块
维护成本 低 低（和 Web 同一份代码） 高（双代码库）

运行时架构

┌──────────────────────────────────────────┐ │ 你的 React App (frontend/) │ │ 86 pages · TanStack Query · CSS Glass │ │ 完全不变，同一份代码 │ └──────────────┬───────────────────────────┘ │ ┌──────────▼──────────┐ │ Capacitor Bridge │ │ JS → Native 翻译层 │ │ Camera · Push · File │ └──────────┬──────────┘ │ ┌──────────▼──────────┐ │ WebView (系统浏览器) │ │ iOS: WKWebView │ │ Android: Chromium │ └──────────┬──────────┘ │ ┌──────────▼──────────┐ │ Native 壳 │ │ .ipa / .aab / .apk │ └──────────────────────┘

本质：Capacitor 不改变你的 UI 代码。你的 React App 跑在系统的 WebView 里，看起来和 Safari/Chrome 打开一模一样。Capacitor 只做两件事：打包成 App + 提供原生 API 桥接。

二、缺陷与风险清单

迁移前必须解决的 8 个问题

HIGH · 必须修复

1. Safe Area 完全缺失

当前 0 处 safe-area-inset-* 引用。iPhone 14 Pro+ 的灵动岛/底部横条会遮挡 UI。需全局 CSS 变量 + 页面级适配。

HIGH · 必须修复

2. Touch 交互依赖 Hover

桌面端使用 :hover 伪类的交互在移动端无反馈。需全局替换为 :active 或 @media (hover: none) 降级。

MED · 建议修复

3. 响应式断点不足

当前 @media 在 420px/520px。现代手机逻辑宽度：iPhone SE=375px, iPhone 14=390px, iPhone 14 Pro Max=430px。需增加 375px/390px 断点。

MED · 建议修复

4. Web Push → Capacitor Push 迁移

sw.js 的 push 事件基于 Web Push API。在 WebView 中不可用。需安装 @capacitor/push-notifications 并适配。

MED · 建议修复

5. 图片上传流程

当前使用 <input type="file"> 或 Multer 上传。在 Capacitor 中需接入 @capacitor/camera 和 @capacitor/filesystem 以支持拍照/相册选择。

LOW · 可后续修复

6. App 图标与启动画面

需要 1024×1024 icon + 各尺寸自适应图标 + 启动画面。Capacitor 有工具自动生成。

LOW · 可后续修复

7. 状态栏样式

iOS 状态栏文字颜色需要配置（深色/浅色）。Capacitor StatusBar 插件 3 行代码搞定。

LOW · 可后续修复

8. 深度链接与 Universal Links

App 内跳转（如推送 → 帖子详情）需要配置 URL Scheme + iOS Associated Domains。Capacitor Deep Links 插件支持。

七、常见问题 FAQ

Capacitor App 会比 React Native App 慢吗？

对于 XMUMDorm 这类内容展示型 App（非游戏/非 AR），不会有可感知的差异。iOS 的 WKWebView 和 Android 的 Chromium WebView 性能都非常好。你的 glass backdrop-filter 在 CSS 中跑 GPU 合成层，比 React Native 的 JS 动画更流畅。

需要改后端吗？

完全不需要。Capacitor 只是一个"浏览器壳"——后端 API 照常响应 HTTP 请求。JWT Token 的存储和发送完全不受影响（localStorage/AsyncStorage → Capacitor Preferences 插件，但 localStorage 本身也能用）。

App Store 审核会拒绝 Capacitor App 吗？

不会。大量知名 App 使用 Capacitor/Cordova 上架（Ionic 自己就是 Capacitor 的创造者，他们大量客户在 App Store）。只要你的 App 提供原生级体验、不是纯网页套壳，审核通过没问题。

用户能感知到这是一个 WebView App 吗？

做好 Safe Area 适配就不会。最大的暴露点是：WebView 中的页面没有 safe-area-inset 适配 → iPhone 灵动岛/底部横条遮挡内容 → 用户立刻知道"这是个网页"。Phase 2 的 Safe Area 适配就是为了解决这个问题。

那 mobile/ 目录怎么办？

保留不删。69 screens + 416 tests + 4 hooks + 15 components 是优质工程资产。未来如果某个功能确实需要原生性能（比如实时视频、AR），可以局部用 RN 实现模块嵌入 Capacitor。现在先让它休眠。
