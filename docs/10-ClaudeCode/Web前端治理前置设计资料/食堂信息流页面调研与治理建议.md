# 食堂信息流页面调研与治理建议

## 范围

本轮治理对象是 Web 的食堂首页（`/eat`）及其进入后的分区、商家、菜品、菜品详情路径。目标不是做外卖下单页，而是让学生能快速完成三件事：找食堂、找店和菜、判断值不值得吃。

已查看的现状：`CanteenHome` 同屏排列搜索、轮播、分区、榜单、随机推荐、吃货广场与快捷入口；桌面端又把它们拆成主列和侧栏。每一块都是独立大卡片，卡片内仍有小卡片、渐变、阴影和圆角，内容优先级相同，浏览起点不明确。

## 一手案例

| 产品 | 可借鉴的页面/资料 | 可复用结论 |
| --- | --- | --- |
| DoorDash | [消费者点餐入口](https://www.doordash.com/food-delivery/) | 首屏先提供位置与搜索，再给有限的横向发现入口；餐厅结果按连续列表阅读，不把每种功能都做成首页模块。 |
| Uber Eats | [消费者点餐入口](https://www.ubereats.com/) | 发现页先解决“在哪里、吃什么”，再用可横向滚动的分类缩小范围；商家卡信息集中在图片、名称、评分和履约信息。 |
| Google Maps Platform | [Place Details](https://developers.google.com/maps/documentation/places/web-service/place-details) | 地点详情的核心字段是名称、地址、营业状态、评分、价格等级、图片与评论；详情页应先给决策信息，再给评论。 |
| Google Maps Platform | [Place Photos](https://developers.google.com/maps/documentation/places/web-service/place-photos) | 照片是地点/商家的一级内容，适合作为商家和菜品卡的主要视觉入口，不能缩成装饰性缩略图。 |
| Yelp | [Places API 概览](https://docs.developer.yelp.com/docs/places-intro) | 本地餐饮信息以搜索、商家信息、评论三段组成；评分和用户评论是判断质量的核心，不需要额外的“功能说明”占位。 |
| Yelp | [Business Search](https://docs.developer.yelp.com/reference/v3_business_search) | 检索结果可围绕名称、地址、类别、评分、评论数、距离、价格组织；字段应服务筛选和比较，而不是全部塞入卡片。 |

注：DoorDash、Uber Eats 的消费者网页会依据地区、登录状态和反爬策略变化；上表对它们的结论来自其官方产品页面的交互结构观察。其余字段结论均可由对应官方文档直接核对。

## 结论

### 桌面三列

页面外层保持全站既有三列：左侧站内导航，中间内容列，右侧全站辅助区。食堂首页不要在中间内容列里再做一个固定侧栏，避免形成“左导航 + 首页主栏 + 首页右栏 + 全站右栏”的四列。

中间列以单一连续信息流为主，最大阅读宽度约 `960px`。需要并排的内容只限于同一类的并列项，例如食堂入口或两张推荐卡；不能让榜单、随机推荐、社区帖各自争抢半屏。

### 首屏和筛选

首屏只保留两件事：搜索框和食堂入口。删除“校园食堂 / 食堂”大标题、入口/榜单/推荐三项统计、轮播图与快捷入口卡。它们没有帮助用户选择餐食，只增加页面开场高度。

搜索框下方用食堂分区作为一级筛选。桌面端显示为紧凑的图标加名称横排或两行网格；选中分区后进入对应商家列表。分类是导航，不需要包成一组厚重卡片。

排序或筛选只在结果页出现：`综合推荐`、`评分最高`、`最近新增` 即可。当前已有的热菜、店铺、新品三类榜单可复用其数据接口，收敛为同一个排序控件，而不是首页独立模块。

### 首页信息流

搜索和分区之后，首页仅保留两个内容段：

1. `大家都在吃`：默认展示 4 到 6 个评分有效的菜品，双列图片卡。
2. `食堂讨论`：展示最近 3 条带图或有互动的食评帖子，连续列表，末尾一个“查看全部”。

“今天吃什么”不再占用独立大区块。将它折叠为 `大家都在吃` 标题旁的骰子图标按钮，点击后在同一区块替换一张菜品卡。现有 `pickRandomMeal` 接口可以直接复用。

轮播图只在有真实活动、营业变更或专题内容时才展示；没有内容时不保留骨架或空白。它不能是首页常驻的装饰性大图。

### 商家和菜品卡

商家卡：横向图片在左，右侧依次为店名、所在食堂、评分和评论数、营业状态。价格、距离等后端没有稳定字段时不伪造。整张卡可点击，卡间使用 `1px` 分割线或很浅的底色，不再每张都投影。

菜品卡：双列布局，图片使用固定比例并占卡片上半部；下方保留菜名、商家、价格、评分。描述最多两行且只在已有内容时出现。图片是主要扫读锚点，评分和价格在同一视觉层级，商家名降低一级。

食评帖：使用左图右文的紧凑行，保留摘要、作者、时间和互动数。删除“吃货广场”下面的解释性提示文案，真实内容就是最好的说明。

### 详情阅读

菜品详情按以下顺序：大图、菜名与价格、商家与食堂位置、评分与评论数、菜品描述、点评列表。收藏、写点评放在标题信息附近；商家资料作为可点击链接，不做额外的快照卡。

商家菜单继续使用现有的“左分类、右菜品列表”模式。这是少数真正需要双栏的页面：分类承担页内定位，右侧承担连续浏览。首页不要复刻这个结构。

### 空状态

空状态只说明当前没有什么，并给出一个直接动作。例如“这个食堂还没有收录店铺”配“查看其他食堂”，或“还没有点评”配“写第一条点评”。不要使用插画、虚构统计或多层卡片。

## 现有页面的治理顺序

1. 先整理 `/eat`：删除首页 hero、快捷入口侧栏；将分区设为首屏导航；移除默认轮播；收敛榜单、随机推荐和吃货广场。
2. 再整理分区商家列表：统一商家卡字段与筛选/排序，确认没有后端数据的字段不展示。
3. 整理商家菜单：保留分类侧栏，统一菜品卡比例、价格和评分的位置。
4. 最后整理菜品详情与点评：先完成信息层级，再处理写点评、收藏和空状态。

## 实施边界

- 不新建接口。首页已使用的 `getRegions`、排行榜接口、`pickRandomMeal`、`getFoodArticles` 都可复用。
- 不为了首页合并而改动商家菜单的分类导航逻辑。
- 不保留没有业务内容的轮播、统计或快捷入口。
- 不新增一套食堂专用组件库；优先复用已有 `FoodCard`、搜索、查询键、全站壳和 token，仅在现有组件无法表达时再抽取。
