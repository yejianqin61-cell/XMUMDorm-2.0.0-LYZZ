# 树洞帖子详情页调研与改进方案

## 结论

帖子详情页应是一条连续的阅读和讨论流，而不是正文卡、操作卡、统计卡、提示卡的集合。桌面端可以有右栏，但右栏只放不与主列重复的相关推荐；作者、点赞、评论数和参与入口都应留在主列各自出现一次。

## 一手资料与可复用结论

| 产品 / 一手来源 | 可复用结论 | 树洞采用方式 |
| --- | --- | --- |
| [GitHub Discussions：About discussions](https://docs.github.com/en/discussions/collaborating-with-your-community-using-discussions/about-discussions) | 讨论以分类承载，置顶、标记回答、关闭等均是按需出现的管理能力；不是常驻的阅读信息。 | 标签留在作者信息旁；举报、删除收进图标菜单或仅对有权限者显示。不要新增“公告模式”“社区讨论”等状态说明。 |
| [GitHub Primer：Typography](https://primer.style/product/ui-architecture/foundations/typography/) | 排版、间距和色彩属于基础 token，正文阅读应优先保证层级和可读性。 | 正文使用现有全站字体栈，桌面端宽度限制在 `680px - 760px`，正文 `16px/1.75`；不为树洞另造字体。 |
| [GitHub Primer：Layout](https://primer.style/product/ui-architecture/layout/) | 页面布局先区分主内容和辅助信息，辅助区不应抢主阅读区。 | `>= 1200px` 保持主列和窄右栏；`< 1200px` 右栏内容下沉或隐藏。主列不因右栏而压缩到难读。 |
| [Discourse：Understanding Discourse for new users](https://meta.discourse.org/t/understanding-discourse-for-new-users/96331) | Topic 由原帖和按时间顺序的回复组成；回复可指向具体帖子，但仍保持在连续讨论中。操作位于每条帖子底部，更多操作收在省略号里。 | 评论采用连续列表和细分割线；一级评论不再单独浮卡。二级回复缩进并用左侧细线关联，最多保留一层。回复、举报、删除按权限显示在评论行内，不做额外卡片。 |
| [Stack Overflow Design System：Typography](https://stackoverflow.design/product/base/typography/) | 信息密度高的社区界面仍依赖固定的字阶和行高，而非用大标题、粗字和阴影制造层级。 | 作者名 `14px/600`，时间、标签、评论操作 `13px`，评论正文 `15px/1.65`；标题仅在帖子确有标题时显示。 |
| [Stack Overflow Design System：Color palette](https://stackoverflow.design/product/base/colors/) | 颜色分为中性色界面层和少量语义强调色；边界用于分区，不能用强阴影堆出层级。 | 页面白底或极浅灰底；正文容器用 `1px` 浅边界，不使用玻璃、背景模糊和大阴影；蓝色只给链接、聚焦和已选状态，点赞用现有强调色。 |

本次没有把 Reddit 作为证据来源：其帮助中心对当前调研环境返回访问限制，不能据此做不可复核的结论。以上来源均为产品方官方文档或官方设计系统。

## 建议布局

```text
返回

作者 · 时间 · 标签                         相关推荐（仅有真实数据时显示）
正文 / 图片
点赞 · 评论数 · 更多

评论（n）
评论流

固定评论输入框
```

`帖子详情`、`社区帖子`、`正文主阅读列`、`快捷操作`、`帖子快照`、`下一步`均不应作为用户可见文案出现。

## 对当前实现的改动建议

当前页面的拥挤来自 [PostDetail.jsx](../../../frontend/src/pages/PostDetail.jsx) 同时渲染了页头统计、正文标签、右侧“快捷操作”、右侧“帖子快照”和“下一步”，而点赞和评论入口在其中重复出现。`PostDetail.css` 还将正文容器做成了 24px 圆角、玻璃背景和大阴影；多图使用 `StackedCardCarousel` 叠卡。

按以下顺序治理：

1. 页头只留返回；保留删帖入口给作者，但改为图标菜单。删除 `eyebrow`、说明文、`pageMeta` 和“写评论”按钮。
2. 正文区保留作者、时间、标签、内容、图片和一行点赞/评论/更多操作。点赞数在此处出现一次；评论数作为跳转到评论区的链接出现一次。
3. 删除 `meta` 中的“快捷操作”卡，删除 `aside` 中的“帖子快照”和“下一步”。后续接入真实相关推荐时，再复用站点已有的推荐组件放到 `aside`。
4. 多图改为普通单图轮播：一张主图、左右箭头、`当前张数 / 总张数`。保留已有预览、拖拽和切换状态，不再显示后方叠卡和卡片阴影。
5. 评论区去卡片化：一级评论之间用分割线；头像缩小；回复只缩进 `24px` 并加左边线。评论操作默认弱化，悬停或键盘聚焦后可见；删除和举报仍保留现有权限逻辑。
6. 视觉收口：正文模块圆角不超过 `8px`，仅保留浅边界；去掉图片氛围模糊层、`backdrop-filter` 和大阴影。沿用现有 Token 和 `DetailPageLayout`，不新建页面模板或数据接口。

## 桌面与窄屏规则

| 宽度 | 布局 |
| --- | --- |
| `>= 1200px` | 内容区最大 `760px`，右侧 `280px - 320px`；右侧只允许相关推荐。 |
| `768px - 1199px` | 单主列；相关推荐下沉到评论后。 |
| `<= 767px` | 单列；保留固定评论输入框，图片全宽，所有次级操作归入更多菜单。 |

## 验收标准

- 首屏只看到返回、作者信息、正文和一个操作行；没有解释页面结构的英文或产品腔文案。
- 点赞、评论数、写评论入口各只出现一次。
- 正文、图片、评论之间没有连续叠放的浮卡和玻璃效果。
- 多图时只显示一张主图；可用箭头、拖拽和键盘切换，并可打开原有图片预览。
- 评论、回复、举报、删除、登录拦截、乐观更新和缓存更新行为与改造前一致。
- `frontend` 的构建和桌面/窄屏视觉回归通过；不改 `frontend-app/`、`shared/` 或后端接口。

## 不在本次范围

- 不增加收藏、转发、投票、已解决答案等后端没有的数据能力。
- 不把 Discourse 的时间线、GitHub 的问答标记照搬到树洞；当前评论量和数据模型不需要这些复杂度。
